# Mini DBMS API Server 구름 과제 명세서

## 1. 문서 목적

이 문서는 `미니 DBMS - API 서버` 과제의 최종 구현 범위를 고정하는 기준 문서다.  
목표는 다음 세 가지를 동시에 만족하는 것이다.

1. 외부 클라이언트가 HTTP API를 통해 SQL 요청을 보낼 수 있어야 한다.
2. 고정 크기 스레드 풀로 요청을 병렬 처리해야 한다.
3. 기존 SQL 처리기와 B+ 트리 인덱스를 최대한 그대로 재사용해야 한다.

본 명세서는 "학습용 최소 구현"을 기준으로 하되, 보고서와 포트폴리오에 넣을 수 있을 정도로 구조와 검증 항목을 명확히 잡는 것을 목표로 한다.

## 2. 최종 결정 요약

| 항목 | 최종 결정 | 선정 이유 |
| --- | --- | --- |
| 통신 방식 | HTTP/1.1 + JSON | 외부 클라이언트 사용성이 높고 `curl` 기반 테스트가 쉽다. |
| 엔드포인트 | `POST /query`, `GET /health` | 기능을 최소화하면서도 DB API 서버의 핵심 흐름을 드러내기 좋다. |
| 요청 형식 | `{"sql":"..."}` | 이후 필드 확장과 에러 처리가 쉽다. |
| SQL 범위 | `INSERT`, `SELECT`만 지원 | 기존 SQL 처리기 범위와 가장 잘 맞고 최소 구현에 적합하다. |
| WHERE 범위 | 단일 조건, `=`만 지원 | 파서와 실행기 구조를 단순하게 유지할 수 있다. |
| 요청당 SQL 수 | 1문장만 허용 | 락 정책, 에러 처리, 성능 측정 기준을 단순하게 유지할 수 있다. |
| 스키마 방식 | `.schema` 파일 사용 | 현재 코드베이스가 이미 이 구조를 사용하고 있어 재사용성이 가장 높다. |
| 타입 시스템 | 이번 버전에서는 제외 | 현재 `.schema`는 컬럼 이름만 가지며 타입 검증을 추가하면 구현 범위가 커진다. |
| 서버 구조 | API 서버와 DB 엔진 모듈 분리 | 네트워크 계층과 SQL 실행 계층의 책임을 분리하기 쉽다. |
| 스레드 처리 | 고정 크기 스레드 풀 + 고정 크기 요청 큐 | 구현이 단순하고 성능 비교 실험에 적합하다. |
| 기본 워커 수 | 4 | 단일 스레드보다 병렬성 확인이 쉽고 과도한 락 경합도 피할 수 있다. |
| 기본 큐 크기 | 64 | burst 요청 흡수와 지연 통제 사이의 균형이 좋다. |
| 과부하 처리 | 큐 포화 시 `503` 반환 | 무한 대기보다 서버 상태를 명확히 전달할 수 있다. |
| 동시성 제어 | 전역 `pthread_rwlock_t` | `SELECT` 병렬성은 확보하고 구현 복잡도는 낮출 수 있다. |
| 응답 형식 | `columns + rows` 구조 | 현재 결과 집합 구조와 직접 매핑되며 C에서 만들기 쉽다. |
| 추가 차별점 | 스레드 수별 성능 측정을 필수 검증 항목으로 포함 | 아키텍처 선택의 효과와 한계를 정량적으로 보여줄 수 있다. |

## 3. 프로젝트 범위

### 3.1 구현 목표

본 프로젝트는 외부 클라이언트가 SQL 문자열을 HTTP로 전송하면, 서버가 이를 내부 DB 엔진으로 전달해 실행하고 JSON 결과를 반환하는 시스템을 구현한다.

### 3.2 범위 선정 이유

과제의 핵심은 "완전한 DBMS"가 아니라 다음 세 가지다.

1. API 서버를 통해 DB 기능을 외부에 노출하는 구조
2. 스레드 풀 기반 병렬 요청 처리
3. 내부 DB 엔진과 API 서버의 연결 설계

따라서 SQL 범위, 스키마 시스템, HTTP 기능은 과제 핵심을 해치지 않는 선에서 축소하고, 대신 동시성, 구조 분리, 검증에 집중한다.

## 4. 지원 기능

### 4.1 SQL 지원 범위

서버는 외부 API 기준으로 아래 SQL만 지원한다.

1. `INSERT`
2. `SELECT`

추가 세부 범위는 아래와 같다.

| 항목 | 지원 여부 | 비고 |
| --- | --- | --- |
| `INSERT INTO table VALUES (...)` | 지원 | 기본 삽입 |
| `INSERT INTO table (col1, col2) VALUES (...)` | 지원 | 컬럼 리스트 선택 가능 |
| `SELECT * FROM table` | 지원 | 전체 컬럼 조회 |
| `SELECT col1, col2 FROM table` | 지원 | 명시 컬럼 조회 |
| `WHERE column = literal` | 지원 | 단일 조건만 허용 |
| 여러 조건 `AND`, `OR` | 미지원 | 최소 구현 제외 |
| 비교 연산 `>`, `<`, `>=`, `<=` | 미지원 | 최소 구현 제외 |
| `UPDATE`, `DELETE`, `JOIN` | 미지원 | 최소 구현 제외 |
| 다중 SQL 요청 | 미지원 | 요청당 1문장만 허용 |

### 4.2 선정 이유

현재 코드베이스의 parser와 executor는 `INSERT`, `SELECT`, `WHERE column = literal` 구조에 가장 잘 맞는다.  
복합 조건, 타입 검증, 추가 DML까지 확장하면 과제의 핵심인 API 서버와 동시성 구조보다 SQL 문법 확장에 시간이 더 많이 들어간다.

## 5. 스키마와 저장 방식

### 5.1 스키마 방식

초기 스키마는 `schema.sql`이 아니라 `.schema` 파일로 준비한다.  
예시는 다음과 같다.

```text
db/users.schema
```

```text
id
name
age
```

규칙은 다음과 같다.

1. 파일명에서 확장자를 제외한 이름이 테이블 이름이다.
2. 각 줄은 컬럼 이름 하나를 뜻한다.
3. 컬럼 순서는 저장 순서이자 `SELECT *`의 출력 순서다.

### 5.2 데이터 저장

실제 레코드 데이터는 기존 DB 엔진이 사용하는 `.data` 파일 구조를 그대로 사용한다.  
API 서버는 저장 포맷을 직접 해석하지 않고 DB 엔진을 통해 접근한다.

### 5.3 타입 시스템

이번 버전에서는 컬럼 타입(`INT`, `TEXT`)을 명세에서 제외한다.  
즉, `.schema`는 컬럼 이름만 정의하고 타입 검증은 수행하지 않는다.

### 5.4 선정 이유

현재 코드베이스는 `.schema` 파일을 기준으로 컬럼 정보를 읽고, parser는 숫자/문자열 리터럴 정도만 구분한다.  
여기에 `schema.sql` 파서나 타입 시스템까지 추가하면 구현 범위가 빠르게 커지므로, 최소 구현에서는 기존 구조를 그대로 사용하는 편이 합리적이다.

## 6. API 명세

### 6.1 기본 규칙

서버는 HTTP/1.1의 최소 기능만 지원한다.

1. 요청당 TCP 연결 1개를 사용한다.
2. 응답 후 연결을 닫는다.
3. `POST /query`는 `Content-Type: application/json`만 허용한다.
4. SQL 문자열 최대 길이는 `4096 bytes`로 제한한다.
5. chunked transfer, keep-alive, multipart는 지원하지 않는다.

### 6.2 선정 이유

과제 핵심은 HTTP 전체 구현이 아니라 DB API 서버 구조다.  
따라서 표준적인 API 사용 경험은 유지하되, HTTP 파싱 범위는 최소한으로 제한해 구현 리스크를 줄인다.

### 6.3 엔드포인트

#### `GET /health`

서버가 정상 동작 중인지 확인하는 엔드포인트다.

응답 예시:

```json
{
  "status": "ok"
}
```

선정 이유:

- 배포 전후 점검과 자동 테스트에 유용하다.
- 구현 비용이 매우 작고 서버 생존 여부를 빠르게 확인할 수 있다.

#### `POST /query`

SQL 실행 요청을 처리하는 핵심 엔드포인트다.

요청 예시:

```json
{
  "sql": "SELECT id, name FROM users WHERE id = 1;"
}
```

선정 이유:

- "외부 클라이언트가 SQL을 보낸다"는 과제 의도를 가장 직접적으로 드러낸다.
- `INSERT`와 `SELECT`를 URL로 분리하지 않아도 되어 API가 단순해진다.

### 6.4 요청/응답 형식

#### `SELECT` 성공 응답

```json
{
  "success": true,
  "type": "select",
  "columns": ["id", "name"],
  "rows": [["1", "kim"]],
  "row_count": 1,
  "used_index": true
}
```

#### `INSERT` 성공 응답

```json
{
  "success": true,
  "type": "insert",
  "affected_rows": 1
}
```

#### 실패 응답

```json
{
  "success": false,
  "error_code": "SQL_PARSE_ERROR",
  "message": "expected FROM after select list"
}
```

### 6.5 응답 형식 선정 이유

`columns + rows` 구조는 현재 DB 엔진의 결과 구조와 직접 연결되며, 컬럼 순서 보장도 쉽다.  
또한 타입 시스템을 도입하지 않는 이번 버전에서는 셀 값을 문자열로 반환하는 편이 구현과 설명 모두 단순하다.

## 7. HTTP 상태 코드 정책

| 상태 코드 | 의미 | 사용 조건 |
| --- | --- | --- |
| `200 OK` | 정상 처리 | `SELECT`, `INSERT`, `health` 성공 |
| `400 Bad Request` | 잘못된 입력 | 잘못된 JSON, `sql` 누락, 빈 SQL, 지원하지 않는 SQL, SQL 문법 오류 |
| `404 Not Found` | 지원하지 않는 경로 | `/health`, `/query` 외 경로 |
| `503 Service Unavailable` | 과부하 | 요청 큐 포화 |
| `500 Internal Server Error` | 내부 오류 | 저장소, 인덱스, 실행기 등 내부 실패 |

### 7.1 에러 코드

서버는 아래 에러 코드를 사용한다.

1. `INVALID_JSON`
2. `MISSING_SQL_FIELD`
3. `EMPTY_QUERY`
4. `UNSUPPORTED_QUERY`
5. `SQL_PARSE_ERROR`
6. `TABLE_NOT_FOUND`
7. `COLUMN_NOT_FOUND`
8. `EXECUTION_ERROR`
9. `QUEUE_FULL`
10. `INTERNAL_ERROR`

### 7.2 선정 이유

HTTP 상태 코드는 큰 범주만 전달하고, 세부 원인은 `error_code`로 구분하는 방식이 가장 단순하다.  
이 구조는 테스트 스크립트 작성도 쉽고, 클라이언트가 처리 분기를 만들기도 쉽다.

## 8. 서버 아키텍처

### 8.1 모듈 구성

시스템은 아래 두 계층으로 분리한다.

1. API Server
2. DB Engine

각 책임은 다음과 같다.

| 모듈 | 책임 |
| --- | --- |
| API Server | 소켓 생성, `accept`, HTTP 파싱, JSON 파싱, 요청 큐 관리, 응답 직렬화 |
| Thread Pool | 워커 생성, 큐 대기, 작업 분배 |
| DB Engine Adapter | SQL 문자열을 기존 lexer/parser/executor/runtime에 연결 |
| Existing DB Engine | SQL 파싱, 실행, 스키마 로드, 저장소 접근, B+ 트리 인덱스 사용 |

### 8.2 워커 처리 흐름

1. 메인 스레드는 클라이언트 연결을 `accept`한다.
2. 연결된 소켓 FD를 요청 큐에 넣는다.
3. 워커 스레드는 큐에서 FD를 가져온다.
4. 워커는 HTTP 요청을 읽고 JSON에서 `sql`을 추출한다.
5. SQL을 lexer/parser로 해석한다.
6. SQL 종류에 따라 read lock 또는 write lock을 획득한다.
7. DB 엔진을 호출해 결과를 받는다.
8. JSON 응답을 전송하고 연결을 닫는다.

### 8.3 요청 처리 전체 과정

요청이 실제로 어떻게 흘러가는지 한 번에 보면 다음과 같다.

```text
Client
  -> TCP connect
  -> HTTP POST /query
  -> JSON body { "sql": "..." }
Main Thread
  -> accept()
  -> enqueue(client_fd)
Worker Thread
  -> dequeue(client_fd)
  -> HTTP request read
  -> JSON parse
  -> SQL extract
  -> tokenize_sql()
  -> parse_statement()
  -> lock acquire (SELECT=read, INSERT=write)
  -> execute_statement()
  -> lock release
  -> ExecResult -> JSON
  -> HTTP response write
  -> close(client_fd)
Client
  -> JSON response receive
```

이 과정을 SQL 종류별로 요약하면 다음과 같다.

1. `SELECT` 요청은 파싱 후 read lock을 잡고 실행한다.
2. `WHERE id = value` 형태면 B+ 트리 인덱스를 사용할 수 있다.
3. `INSERT` 요청은 write lock을 잡고 `.data` append와 인덱스 갱신을 수행한다.
4. 결과는 `ExecResult`에서 JSON 응답으로 변환된다.

### 8.4 서버 시작 과정

서버가 실행될 때의 초기화 순서는 아래와 같이 정의한다.

```text
process start
  -> CLI 옵션 파싱
  -> DB 디렉터리 확인
  -> 전역 RWLock 초기화
  -> 요청 큐 초기화
  -> Worker Thread 생성
  -> listen socket 생성
  -> bind()
  -> listen()
  -> accept loop 진입
```

구체적인 순서는 다음과 같다.

1. 실행 옵션에서 포트, DB 디렉터리, worker 수, queue 크기를 읽는다.
2. DB 디렉터리 존재 여부와 기본 스키마 파일을 확인한다.
3. 전역 `pthread_rwlock_t`를 초기화한다.
4. 요청 큐와 큐 보호용 mutex/condition variable을 초기화한다.
5. 고정 개수 worker thread를 생성한다.
6. 서버 소켓을 만들고 `bind`, `listen`을 수행한다.
7. 메인 스레드는 `accept loop`에 진입한다.

이 순서를 따르는 이유는 다음과 같다.

1. 요청을 받기 전에 worker와 queue가 준비되어 있어야 한다.
2. 락과 큐 초기화가 끝나기 전에 `accept`를 시작하면 초기 요청 처리 중 오류가 날 수 있다.
3. 초기화 실패 시 어느 단계에서 중단되었는지 명확히 정리하기 쉽다.

### 8.5 서버 종료 과정

서버 종료는 자원 정리와 worker 종료가 꼬이지 않도록 아래 순서를 따른다.

```text
shutdown signal
  -> accept loop 종료
  -> listen socket close
  -> queue shutdown 표시
  -> sleeping worker wake-up
  -> worker join
  -> queue / lock / context destroy
  -> process exit
```

구체적인 종료 절차는 다음과 같다.

1. `SIGINT` 또는 종료 요청을 받으면 신규 요청 수락을 중지한다.
2. `listen_fd`를 닫아 `accept` 루프를 빠져나오게 한다.
3. 요청 큐를 shutdown 상태로 바꾼다.
4. condition variable을 broadcast하여 대기 중인 worker를 깨운다.
5. worker thread는 남은 작업이 없으면 루프를 종료한다.
6. 메인 스레드는 모든 worker를 `join`한다.
7. 큐, 락, 소켓, 서버 컨텍스트를 순서대로 해제한다.

이 과정을 명세에 포함하는 이유는 다음과 같다.

1. 스레드 풀 구조에서는 종료 순서가 잘못되면 worker가 영원히 잠들 수 있다.
2. 소켓과 락, 큐 자원을 어떤 순서로 정리하는지 미리 정의해야 구현이 흔들리지 않는다.
3. graceful shutdown을 명세에 포함하면 테스트와 발표에서 완성도가 높아진다.

### 8.6 동시 요청 예시

아래 예시는 이 서버가 실제로 병렬 요청을 어떻게 처리하는지 설명하기 위한 대표 시나리오다.

#### 예시 1. `SELECT` + `SELECT`

```text
Client A -> POST /query (SELECT * FROM users WHERE id = 1;)
Client B -> POST /query (SELECT name FROM users WHERE id = 2;)

Main Thread
  -> A 연결 enqueue
  -> B 연결 enqueue

Worker 1
  -> A 요청 dequeue
  -> read lock 획득
  -> SELECT 실행

Worker 2
  -> B 요청 dequeue
  -> read lock 획득
  -> SELECT 실행
```

의미:

1. 두 요청 모두 `SELECT`이므로 read lock을 동시에 잡을 수 있다.
2. 따라서 읽기 요청끼리는 병렬성이 살아난다.
3. `WHERE id = value` 조건이면 각 worker가 인덱스를 사용할 수 있다.

#### 예시 2. `INSERT` + `SELECT`

```text
Client A -> POST /query (INSERT INTO users VALUES (...);)
Client B -> POST /query (SELECT * FROM users WHERE id = 10;)

Worker 1
  -> A 요청 dequeue
  -> write lock 획득
  -> INSERT 실행

Worker 2
  -> B 요청 dequeue
  -> read lock 대기
  -> Worker 1의 write lock 해제 후 SELECT 실행
```

의미:

1. `INSERT`는 write lock을 사용하므로 실행 중에는 다른 읽기/쓰기 요청이 대기할 수 있다.
2. 이 구조는 정합성은 단순하게 보장하지만 write-heavy workload에서는 병목이 될 수 있다.
3. 바로 이 지점을 성능 측정 항목에서 검증한다.

### 8.7 이 과정을 명세에 포함하는 이유

이 섹션은 단순 설명용이 아니라, 아래 네 가지를 한 번에 연결해 준다.

1. API 서버가 어디까지 담당하는지
2. Thread Pool과 Queue가 어디서 개입하는지
3. 락이 정확히 어느 시점에 걸리는지
4. 기존 DB 엔진이 어디서 재사용되는지

즉, "돌아가는 과정"을 넣으면 구현 범위와 책임 분리가 독자 입장에서 훨씬 명확해진다.

### 8.8 선정 이유

메인 스레드는 가볍게 연결 수락과 큐 적재만 담당하고, 실제 요청 해석과 SQL 실행은 워커에 넘기는 구조가 가장 단순하다.  
이 방식은 병렬 요청 처리 흐름을 설명하기 쉽고, 스레드 풀 구조도 분명하게 드러난다.

## 9. 스레드 풀과 요청 큐 정책

### 9.1 기본 정책

| 항목 | 결정 |
| --- | --- |
| 워커 수 | 기본값 `4` |
| 요청 큐 크기 | 기본값 `64` |
| 큐 구조 | 고정 크기 원형 큐 |
| 큐 포화 시 동작 | 즉시 `503` 응답 |
| 설정 변경 | 실행 옵션으로 변경 가능 |

### 9.2 기본 실행 예시

```bash
./mini_db_api_server --port 8080 --db-dir ./db --workers 4 --queue-size 64
```

### 9.3 선정 이유

#### 워커 수 4개

- `1개`보다 병렬성 효과를 보여주기 쉽다.
- `8개 이상`보다 락 경합과 디버깅 부담이 작다.
- 현재 구조는 전역 `RWLock` 기반이므로, 너무 많은 워커는 쓰기 workload에서 이득보다 비용이 커질 수 있다.

#### 큐 크기 64

- 순간적인 요청 증가를 어느 정도 흡수할 수 있다.
- 너무 작은 큐보다 burst 대응력이 좋다.
- 너무 큰 큐보다 장시간 대기 요청 누적을 막기 쉽다.

#### 큐 포화 시 503

- 서버가 바쁜 상태를 명확히 드러낸다.
- 무한 대기보다 시스템 상태를 제어하기 쉽다.
- 부하 테스트에서 과부하 시점이 뚜렷하게 보인다.

## 10. 동시성 제어 정책

### 10.1 락 정책

서버는 전역 `pthread_rwlock_t` 하나를 사용한다.

| SQL 종류 | 락 종류 |
| --- | --- |
| `SELECT` | read lock |
| `INSERT` | write lock |

### 10.2 락 범위

락은 DB 엔진 실행 구간에만 적용한다.

1. HTTP 요청 읽기
2. JSON 파싱
3. SQL 토큰화와 파싱
4. read/write lock 획득
5. DB 엔진 실행
6. lock 해제
7. JSON 응답 작성 및 전송

### 10.3 선정 이유

전역 `RWLock`은 현재 과제에서 가장 현실적인 절충안이다.

- 전역 `mutex`보다 `SELECT` 병렬성을 확보할 수 있다.
- 테이블 단위 락이나 인덱스 단위 락보다 구현이 훨씬 단순하다.
- B+ 트리와 저장소에 대한 동시 접근을 안전하게 보호할 수 있다.

이 정책은 성능적으로 완벽하지 않지만, 바로 그 점이 이번 과제의 분석 포인트가 된다.  
특히 `INSERT`가 많아질수록 write lock 때문에 병렬성이 제한되는 모습을 정량적으로 보여줄 수 있다.

## 11. DB 엔진 연동 정책

### 11.1 연동 방식

API 서버는 SQL을 직접 실행하지 않고, 기존 DB 엔진을 감싸는 adapter 계층을 통해 아래 흐름으로 호출한다.

1. SQL 문자열 입력
2. `tokenize_sql`
3. `parse_statement`
4. `execute_statement`
5. `ExecResult`를 JSON으로 직렬화

### 11.2 B+ 트리 사용

`SELECT ... WHERE id = value` 형태의 질의는 가능하면 기존 B+ 트리 인덱스를 사용한다.  
이를 클라이언트가 확인할 수 있도록 `SELECT` 응답에 `used_index` 필드를 포함한다.

### 11.3 선정 이유

이번 과제는 "새 DB 엔진 제작"이 아니라 "기존 엔진의 API 서버화"에 가깝다.  
따라서 기존 구조를 최대한 유지하고, 서버 계층은 입출력과 동시성 문제에 집중하는 것이 맞다.

## 12. 입력 제한과 예외 처리

### 12.1 입력 제한

1. 요청 본문은 JSON이어야 한다.
2. `sql` 필드는 필수다.
3. `sql` 문자열은 비어 있을 수 없다.
4. SQL은 1문장만 허용한다.
5. SQL 길이는 `4096 bytes` 이하여야 한다.

### 12.2 엣지 케이스

서버는 아래 상황을 명시적으로 처리해야 한다.

1. 빈 JSON 또는 잘못된 JSON
2. `sql` 필드 누락
3. 빈 SQL 문자열
4. 지원하지 않는 SQL 종류
5. 없는 테이블 조회
6. 없는 컬럼 조회
7. 큐가 가득 찬 상태에서 새 요청 유입
8. 서버 종료 중 새 연결 유입

### 12.3 선정 이유

학습용 프로젝트라도 입력 제한을 명확히 두지 않으면 HTTP 서버와 DB 엔진 사이 경계가 흐려진다.  
이번 명세는 "무엇을 허용하고 무엇을 거절하는지"를 명확히 하여 테스트 가능성을 높이는 방향을 택한다.

## 13. 테스트 전략

### 13.1 단위 테스트

기존 엔진 테스트는 그대로 유지하고, 서버 계층에 대해 아래 항목을 추가 검증한다.

1. 요청 큐 push/pop 동작
2. 큐 full/empty 처리
3. HTTP 요청 파싱 보조 함수
4. JSON 요청 검증 보조 함수

### 13.2 기능 테스트

`curl` 기반으로 아래 시나리오를 검증한다.

1. `GET /health` 정상 응답
2. `INSERT` 성공
3. `SELECT *` 성공
4. `SELECT ... WHERE id = ?` 성공
5. 잘못된 JSON 요청
6. 잘못된 SQL 요청
7. 없는 테이블 조회

### 13.3 동시성 테스트

병렬 `INSERT`, 병렬 `SELECT`, 혼합 요청을 동시에 보내고 아래를 확인한다.

1. 서버가 종료되지 않는가
2. 응답이 누락되지 않는가
3. B+ 트리와 저장소가 깨지지 않는가

### 13.4 선정 이유

함수 단위 검증만으로는 과제 핵심인 API 서버 구조를 증명하기 어렵고, 통합 테스트만으로는 원인 추적이 어렵다.  
따라서 단위 테스트와 기능 테스트, 동시성 테스트를 모두 포함하는 구성이 적절하다.

## 14. 필수 성능 검증 항목

### 14.1 목적

스레드 풀 크기에 따라 처리 시간이 실제로 어떻게 변하는지 측정하고, 전역 `RWLock` 구조의 특성을 분석한다.

### 14.2 측정 조건

| 항목 | 결정 |
| --- | --- |
| 스레드 수 | `1`, `2`, `4`, `8` |
| 반복 횟수 | 각 조건당 `3회` |
| workload 1 | `SELECT only` |
| workload 2 | `INSERT only` |
| workload 3 | `SELECT + INSERT mixed` |
| 측정 지표 | 총 수행 시간, 평균 응답 시간, 초당 처리량 |

### 14.3 예상 분석 포인트

1. `SELECT only`는 read lock 공유 덕분에 워커 수 증가 효과가 비교적 잘 보일 가능성이 높다.
2. `INSERT only`는 write lock 때문에 워커 수가 늘어도 성능 향상이 제한될 수 있다.
3. mixed workload는 읽기/쓰기 비율에 따라 병목이 어떻게 바뀌는지 보여준다.

### 14.4 선정 이유

이 항목은 단순한 부가 기능이 아니라, 현재 아키텍처 선택의 타당성과 한계를 설명하는 핵심 검증이다.  
결과가 좋아도 의미 있고, 나빠도 왜 그런지 설명할 수 있기 때문에 보고서 품질을 높이는 데 직접 도움이 된다.

## 16. 완료 기준

아래 조건을 만족하면 본 과제의 최소 구현이 완료된 것으로 본다.
