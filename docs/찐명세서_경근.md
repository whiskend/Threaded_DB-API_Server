아래는 **Codex가 그대로 읽고 구현할 수 있도록 작성한 상세 Spec**이다.
핵심 방향은 “기존 CLI SQL 처리기를 갈아엎지 않고, 그 위에 HTTP API 서버 + Thread Pool을 얹는 것”이다.

첨부된 기존 프로젝트 스펙 기준으로 보면 현재 프로젝트의 실제 핵심은 HTTP 서버가 아니라 `sql_processor`라는 C 기반 CLI SQL 처리기이며, SQL 파일을 읽어 lexer/parser/executor를 거쳐 `.schema`, `.data`, in-memory B+Tree를 사용하는 구조다. 
따라서 이번 구현은 **기존 DB 엔진을 보존하고 API 서버 레이어를 추가하는 작업**으로 정의한다.

# Codex용 프로젝트 명세서

# Mini DBMS - Threaded HTTP API Server

## 0. 구현 목표 요약

현재 프로젝트는 다음 기능을 가진 C 기반 미니 SQL 처리기이다.

```text
SQL file
  -> lexer
  -> parser
  -> executor
  -> schema / storage / B+Tree
  -> stdout 출력
```

이번 과제에서는 이를 다음 구조로 확장한다.

```text
HTTP Client
  -> HTTP API Server
  -> Thread Pool
  -> Task Queue
  -> 기존 DB Engine
  -> schema / storage / B+Tree
  -> JSON Response
```

Codex는 기존 CLI 실행 파일인 `sql_processor`를 유지하면서, 새로운 실행 파일 `mini_db_server`를 추가 구현해야 한다.

# 5. 전체 서버 아키텍처

## 5.1 전체 흐름

```text
Client
  |
  | HTTP Request
  v
server.c
  |
  | accept()
  v
TaskQueue
  |
  | client_fd 저장
  v
Worker Thread
  |
  | HTTP request read / parse
  v
POST /query
  |
  | SQL body 추출
  v
db_api.c
  |
  | tokenize_sql()
  | parse_next_statement()
  | execute_statement()
  v
기존 DB Engine
  |
  | ExecResult
  v
JSON Response
```

## 5.2 중요한 설계 결정

Main Thread는 SQL을 읽거나 실행하지 않는다.

Main Thread는 다음만 한다.

```text
1. socket 생성
2. bind
3. listen
4. accept
5. client_fd를 Thread Pool에 submit
```

Worker Thread는 다음을 한다.

```text
1. TaskQueue에서 client_fd pop
2. HTTP 요청 read
3. method/path/body 파싱
4. endpoint별 처리
5. DB 실행
6. HTTP response write
7. close(client_fd)
```

최소 구현 API는 두 개다.

선택 구현으로 `GET /stats`를 추가할 수 있지만, 필수는 아니다.

## 6.1 GET /health

서버 상태 확인용 API이다.

### Request

```http

Host: localhost:8080
```

### Response

```http
HTTP/1.1 200 OK
Content-Type: application/json
Connection: close
```

Body:

```json
{
  "status": "ok",
  "message": "server is running"
}
```

curl 테스트:

```sh
curl http://localhost:8080/health
```

---

## 6.2 POST /query

SQL 실행 API이다.

### Request

```http
POST /query HTTP/1.1
Host: localhost:8080
Content-Type: text/plain
Content-Length: ...
```

Body:

```sql
INSERT INTO users VALUES ('kim', 25);
```

또는:

```sql
SELECT * FROM users WHERE id = 1;
```

주의: 기존 프로젝트의 `id` 컬럼 정책상, `id` 컬럼이 있는 테이블에서는 사용자가 `id` 값을 직접 넣으면 안 된다.
따라서 `users.schema`가 다음과 같다면:

```text
id
name
age
```

올바른 INSERT는 다음이다.

```sql
INSERT INTO users VALUES ('kim', 25);
```

틀린 INSERT는 다음이다.

```sql
INSERT INTO users VALUES (1, 'kim', 25);
```

위 SQL은 explicit id insert이므로 에러가 나야 한다.

# 7. HTTP 파서 최소 명세

완전한 HTTP parser를 만들 필요는 없다.

다음만 처리한다.

## 7.1 지원하는 HTTP 기능

```text
HTTP/1.1 request line
Method
Path
Content-Length
Body
```

## 7.2 지원하지 않는 HTTP 기능

```text
keep-alive
chunked transfer encoding
multipart
query string parsing
URL decoding
header 전체 정규화
HTTP/2
```

## 7.3 Request parsing 규칙

HTTP 요청은 다음 순서로 읽는다.

```text
1. recv()로 header를 읽는다.
2. "\r\n\r\n" 위치를 찾는다.
3. 첫 줄에서 method와 path를 추출한다.
4. POST 요청이면 Content-Length를 찾는다.
5. Content-Length만큼 body를 끝까지 읽는다.
6. body를 null-terminated C string으로 만든다.
```

## 7.4 제한값

다음 상수를 둔다.

```c
#define HTTP_MAX_HEADER_SIZE 8192
#define HTTP_MAX_BODY_SIZE   65536
#define HTTP_MAX_REQUEST_SIZE (HTTP_MAX_HEADER_SIZE + HTTP_MAX_BODY_SIZE)
```

처리 규칙:

| 상황                       | 응답                                                         |
| ------------------------ | ---------------------------------------------------------- |
| header가 8192 bytes 초과    | `431 Request Header Fields Too Large` 또는 `400 Bad Request` |
| body가 65536 bytes 초과     | `413 Payload Too Large`                                    |
| POST인데 Content-Length 없음 | `400 Bad Request`                                          |
| Content-Length가 숫자가 아님   | `400 Bad Request`                                          |
| body가 비어 있음              | `400 Bad Request`                                          |
| 알 수 없는 path              | `404 Not Found`                                            |
| `/query`에 GET 사용         | `405 Method Not Allowed`                                   |

# 8. HTTP 응답 명세

모든 응답은 JSON body를 가진다.

기본 형식:

```http
HTTP/1.1 <status>
Content-Type: application/json
Content-Length: <body length>
Connection: close

<body>
```

응답 전송은 반드시 partial write를 고려해야 한다.

즉, `write()` 또는 `send()`를 한 번 호출하고 끝내면 안 된다.

다음 helper를 구현한다.

```c
int write_all(int fd, const char *buf, size_t len);
```

동작:

```text
while len > 0:
  n = send(fd, buf + sent, len - sent, 0)
  EINTR이면 재시도
  n <= 0이면 실패
```

SIGPIPE 방지를 위해 서버 시작 시 다음 중 하나를 적용한다.

```c
signal(SIGPIPE, SIG_IGN);
```

또는 `send()`에서 `MSG_NOSIGNAL`을 사용한다.
단, portability를 위해 `signal(SIGPIPE, SIG_IGN)` 방식이 더 단순하다.

# 9. POST /query JSON 응답 형식

## 9.1 성공 응답

성공 시 HTTP status는 `200 OK`이다.

응답 body:

```json
{
  "status": "success",
  "results": [
    {
      "type": "insert",
      "affected_rows": 1,
      "generated_id": 1
    }
  ]
}
```

SELECT 예시:

```json
{
  "status": "success",
  "results": [
    {
      "type": "select",
      "used_index": true,
      "columns": ["id", "name", "age"],
      "rows": [
        ["1", "kim", "25"]
      ],
      "row_count": 1
    }
  ]
}
```

복수 statement 예시:

Request body:

```sql
INSERT INTO users VALUES ('kim', 25);
SELECT * FROM users WHERE id = 1;
```

Response body:

```json
{
  "status": "success",
  "results": [
    {
      "type": "insert",
      "affected_rows": 1,
      "generated_id": 1
    },
    {
      "type": "select",
      "used_index": true,
      "columns": ["id", "name", "age"],
      "rows": [
        ["1", "kim", "25"]
      ],
      "row_count": 1
    }
  ]
}
```

## 9.2 실패 응답

실패 시 HTTP status는 상황에 따라 다르다.

| 실패 유형          | HTTP status |
| -------------- | ----------: |
| 잘못된 HTTP 요청    |         400 |
| 잘못된 SQL 문법     |         400 |
| 지원하지 않는 path   |         404 |
| 지원하지 않는 method |         405 |
| body 너무 큼      |         413 |
| 내부 실행 오류       |         500 |

SQL 에러 응답 body:

```json
{
  "status": "error",
  "code": "PARSE_ERROR",
  "message": "expected FROM after SELECT projection",
  "results": []
}
```

중간 statement까지 성공 후 뒤 statement에서 실패하면 rollback하지 않는다.

예시:

Request:

```sql
INSERT INTO users VALUES ('kim', 25);
SELCT * FROM users;
```

Response:

```json
{
  "status": "error",
  "code": "PARSE_ERROR",
  "message": "unexpected token",
  "results": [
    {
      "type": "insert",
      "affected_rows": 1,
      "generated_id": 1
    }
  ]
}
```

중요: 기존 CLI와 마찬가지로 transaction이 없으므로 앞에서 성공한 INSERT는 `.data`에 남아도 된다.

# 10. DB API Adapter 명세

신규 파일:

```text
include/db_api.h
src/db_api.c
```

## 10.1 목적

기존 CLI 실행 흐름을 HTTP 서버에서 재사용하기 위한 adapter이다.

CLI는 stdout에 table을 출력하지만, API 서버는 JSON 문자열을 반환해야 한다.

따라서 `db_api.c`는 다음 역할을 가진다.

```text
SQL string
  -> tokenize_sql()
  -> parse_next_statement()
  -> execute_statement()
  -> ExecResult
  -> JSON string
```

## 10.2 구조체

```c
typedef struct {
    ExecutionContext ctx;
    pthread_mutex_t db_mutex;
} DbApi;
```

## 10.3 함수

```c
int db_api_init(DbApi *api, const char *db_dir);
void db_api_destroy(DbApi *api);

int db_api_execute_sql(
    DbApi *api,
    const char *sql,
    char **out_json,
    int *out_http_status
);
```

규칙:

* `out_json`은 heap string이다.
* caller가 `free(*out_json)` 해야 한다.
* 성공 시 `STATUS_OK` 반환.
* 실패 시 기존 `StatusCode` 계열 값을 반환한다.
* `out_http_status`에는 HTTP status code를 저장한다.

## 10.4 동시성 규칙

기존 `ExecutionContext`는 thread-safe하지 않다.

따라서 최소 구현에서는 DB 실행 전체를 하나의 mutex로 보호한다.

```c
pthread_mutex_lock(&api->db_mutex);

/*
 tokenize_sql()
 parse_next_statement()
 execute_statement()
 ExecResult -> JSON
*/

pthread_mutex_unlock(&api->db_mutex);
```

이 방식은 DB 작업 자체는 한 번에 하나씩 실행되지만, 다음 장점이 있다.

```text
1. B+Tree mutation 보호
2. TableRuntime cache 보호
3. .data append race condition 방지
4. auto-id next_id race condition 방지
5. 구현 난이도 최소화
```

발표 시에는 다음과 같이 설명한다.

```text
Thread Pool로 여러 client connection을 동시에 처리하되,
기존 DB 엔진의 ExecutionContext와 B+Tree가 thread-safe하지 않기 때문에
DB critical section은 Global Mutex로 보호했습니다.
```

# 11. JSON Writer 명세

신규 파일:

```text
include/json_writer.h
src/json_writer.c
```

## 11.1 목적

외부 JSON 라이브러리를 쓰지 않고 안전한 JSON 문자열을 생성한다.

외부 dependency를 추가하지 않는다.

## 11.2 필수 기능

다음 문자는 반드시 escape한다.

| 원본              | JSON escape |
| --------------- | ----------- |
| `"`             | `\"`        |
| `\`             | `\\`        |
| newline         | `\n`        |
| carriage return | `\r`        |
| tab             | `\t`        |
| control char    | `\u00XX`    |

## 11.3 동적 문자열 버퍼

다음 구조체를 구현한다.

```c
typedef struct {
    char *data;
    size_t len;
    size_t cap;
} JsonBuffer;
```

함수:

```c
int json_buffer_init(JsonBuffer *buf);
void json_buffer_free(JsonBuffer *buf);
int json_append(JsonBuffer *buf, const char *text);
int json_append_char(JsonBuffer *buf, char ch);
int json_append_escaped_string(JsonBuffer *buf, const char *text);
char *json_buffer_take(JsonBuffer *buf);
```

`json_buffer_take()`는 내부 `data` 소유권을 caller에게 넘긴다.

# 12. ExecResult -> JSON 변환 규칙

기존 `ExecResult`를 JSON으로 변환한다.

## 12.1 INSERT

기존 `ExecResult`:

```c
type = RESULT_INSERT
affected_rows = 1
has_generated_id = 1 또는 0
generated_id = ...
```

JSON:

```json
{
  "type": "insert",
  "affected_rows": 1,
  "generated_id": 1
}
```

`has_generated_id == 0`이면 `generated_id` 필드는 생략한다.

## 12.2 SELECT

기존 `ExecResult`:

```c
type = RESULT_SELECT
query_result.columns
query_result.rows
used_index
affected_rows
```

JSON:

```json
{
  "type": "select",
  "used_index": true,
  "columns": ["id", "name", "age"],
  "rows": [
    ["1", "kim", "25"]
  ],
  "row_count": 1
}
```

주의:

* 모든 cell value는 문자열로 반환한다.
* 기존 storage가 타입 시스템을 갖지 않기 때문에 number도 문자열이다.
* `used_index`는 발표와 테스트에 유용하므로 반드시 포함한다.

# 13. Thread Pool 명세

신규 파일:

```text
include/thread_pool.h
src/thread_pool.c
include/task_queue.h
src/task_queue.c
```

## 13.1 Task 구조

최소 구현에서는 task에 client socket fd만 저장한다.

```c
typedef struct {
    int client_fd;
} Task;
```

## 13.2 Task Queue

bounded circular queue를 구현한다.

```c
#define TASK_QUEUE_CAPACITY 1024
```

구조체:

```c
typedef struct {
    Task items[TASK_QUEUE_CAPACITY];
    size_t head;
    size_t tail;
    size_t count;
    int shutdown;
    pthread_mutex_t mutex;
    pthread_cond_t not_empty;
    pthread_cond_t not_full;
} TaskQueue;
```

함수:

```c
int task_queue_init(TaskQueue *queue);
void task_queue_destroy(TaskQueue *queue);

int task_queue_push(TaskQueue *queue, Task task);
int task_queue_pop(TaskQueue *queue, Task *out_task);

void task_queue_shutdown(TaskQueue *queue);
size_t task_queue_size(TaskQueue *queue);
```

동작:

* `push()`는 queue가 가득 차면 `not_full`을 기다린다.
* `pop()`은 queue가 비어 있으면 `not_empty`를 기다린다.
* `shutdown`이 켜지면 대기 중인 thread를 깨운다.
* shutdown 상태에서 `pop()`은 실패를 반환하여 worker loop를 종료할 수 있게 한다.

## 13.3 Thread Pool 구조체

```c
typedef void (*TaskHandler)(Task task, void *user_data);

typedef struct {
    pthread_t *threads;
    size_t thread_count;
    TaskQueue queue;
    TaskHandler handler;
    void *user_data;
    int started;
} ThreadPool;
```

함수:

```c
int thread_pool_init(
    ThreadPool *pool,
    size_t thread_count,
    TaskHandler handler,
    void *user_data
);

int thread_pool_submit(ThreadPool *pool, Task task);

void thread_pool_shutdown(ThreadPool *pool);
void thread_pool_destroy(ThreadPool *pool);
```

## 13.4 Worker 동작

```text
while true:
  task_queue_pop()
  if shutdown:
    break
  handler(task, user_data)
```

서버에서는 handler가 client request 전체를 처리한다.

# 17. 동시성 설계

## 17.1 Thread Pool 동시성

여러 client connection은 worker thread가 동시에 처리한다.

```text
client A -> worker 1
client B -> worker 2
client C -> worker 3
client D -> worker 4
```

## 17.2 DB Critical Section

기존 DB 엔진은 thread-safe하지 않다.

특히 다음이 공유 mutable state다.

```text
ExecutionContext.tables
TableRuntime.next_id
TableRuntime.id_index
B+Tree nodes
.data append
```

따라서 `db_api_execute_sql()` 내부에서 global mutex를 사용한다.

```text
worker 1: HTTP read 가능
worker 2: HTTP read 가능
worker 3: HTTP read 가능

DB 실행:
  오직 하나의 worker만 db_mutex를 잡고 실행
```

## 17.3 최소 구현에서 허용되는 한계

이 방식은 진짜 DB 실행을 병렬화하지 않는다.

하지만 과제에서 중요한 세 요소는 충족한다.

```text
1. Thread Pool이 있다.
2. 요청은 worker thread에 할당된다.
3. 멀티스레드 동시성 이슈를 인식하고 lock으로 보호한다.
```

README에는 이 한계를 명확히 적는다.

```text
현재 구현은 안전성을 위해 DB Engine 전체를 Global Mutex로 보호한다.
따라서 worker thread들은 요청을 병렬로 수신하고 처리하지만,
실제 DB Engine 접근은 한 번에 하나씩 수행된다.
향후 개선으로 pthread_rwlock_t를 사용하여 SELECT는 concurrent read,
INSERT는 exclusive write로 분리할 수 있다.
```

# 23. 메모리 관리 규칙

# 28. Codex에게 전달할 최종 작업 지시

아래 문단을 Codex에게 그대로 전달하면 된다.

```text
현재 프로젝트는 C 기반 CLI SQL Processor이며, sql_processor 실행 파일을 통해 SQL 파일을 읽고
lexer/parser/executor/storage/B+Tree를 사용해 INSERT/SELECT를 처리한다.

이제 기존 기능을 보존하면서 mini_db_server라는 HTTP API 서버 실행 파일을 추가하라.

필수 요구사항:
- 기존 sql_processor와 기존 테스트는 깨지면 안 된다.
- mini_db_server -d <db_dir> -p <port> -t <threads> 형태로 실행되게 하라.
- GET /health는 JSON으로 서버 상태를 반환하라.
- POST /query는 request body의 raw SQL string을 기존 DB 엔진으로 실행하고 JSON 결과를 반환하라.
- Request body는 text/plain SQL로 취급하라. JSON request parsing은 구현하지 마라.
- Thread Pool과 Task Queue를 구현하라.
- Main thread는 accept만 하고 client_fd를 Task Queue에 넣어라.
- Worker thread가 HTTP request read/parse/execute/respond/close를 담당하게 하라.
- 기존 ExecutionContext는 thread-safe하지 않으므로 DbApi 내부에 Global Mutex를 두고 DB 실행 전체를 보호하라.
- ExecResult를 JSON으로 변환하라.
- SELECT 결과에는 used_index, columns, rows, row_count를 포함하라.
- INSERT 결과에는 affected_rows와 generated_id가 있으면 generated_id를 포함하라.
- 잘못된 SQL, 잘못된 HTTP 요청, body too large, unknown path 등에 대해 JSON error response를 반환하라.
- make test에 API 통합 테스트를 추가하라.
- README에 실행 방법, API 명세, Thread Pool 구조, 동시성 제어, 데모 명령어를 작성하라.
```
