아래는 **Codex에게 그대로 전달할 수 있는 단계별 작업 지시서**다.
이번 버전은 단순 프롬프트가 아니라, **우리가 읽고도 Codex가 어떤 파일을 만들고, 어떤 함수가 어떤 책임을 갖는지 바로 이해할 수 있도록** 함수 단위까지 쪼갠 상세 명세다.

기준은 다음이다.

* 기존 `sql_processor`는 그대로 유지한다.
* 새 실행 파일 `mini_db_server`를 추가한다.
* 통신은 **HTTP/JSON**으로 한다.
* 요청 처리는 **Thread Pool + Bounded Queue**로 한다.
* DB 동시성 제어는 **`pthread_rwlock_t`**로 한다.
* SELECT는 read lock, INSERT 포함 요청은 write lock으로 처리한다.
* 단, SELECT도 lazy runtime loading 때문에 사전 preload 단계가 필요하다.
* 기존 프로젝트는 현재 CLI SQL Processor이며, `ExecutionContext`, `.data` append, B+Tree mutation은 thread-safe하지 않으므로 API 서버 추가 시 lock 전략이 필요하다.

# 4. 서버 실행 옵션 명세

## 4.1 실행 예시

```bash
./mini_db_server -d db -p 8080 -t 4
```

또는:

```bash
./mini_db_server --db db --port 8080 --threads 4
```

## 4.2 옵션

| 옵션                | 의미              | 필수 여부 |                         기본값 |
| ----------------- | --------------- | ----: | --------------------------: |
| `-d`, `--db`      | DB 디렉터리         |    필수 |                          없음 |
| `-p`, `--port`    | 서버 포트           |    선택 |                      `8080` |
| `-t`, `--threads` | Worker Thread 수 |    선택 | CPU core 수 × 2, 최소 4, 최대 32 |
| `-h`, `--help`    | 도움말 출력          |    선택 |                          없음 |

## 4.3 동작 규칙

* `-d`가 없으면 서버를 실행하지 않고 에러 출력 후 종료한다.
* `-p`가 없으면 `8080`을 사용한다.
* `-t`가 없으면 기본 thread 수를 계산한다.
* `-t`가 1보다 작거나 비정상 값이면 에러 처리한다.
* 너무 큰 thread 수는 32로 제한한다.

### `int server_run(Server *server)`

#### 인자

```c
Server *server
```

#### 동작

accept loop를 실행한다.

Pseudo code:

```c
while (1) {
    int client_fd = accept(server->listen_fd, ...);

    if (client_fd < 0) {
        if (errno == EINTR) {
            continue;
        }
        continue;
    }

    Task task;
    task.client_fd = client_fd;

    int rc = thread_pool_try_submit(&server->pool, task);

    if (rc != 0) {
        stats_inc_rejected(&server->stats);
        http_send_error_response(
            client_fd,
            503,
            "QUEUE_FULL",
            "server task queue is full"
        );
        close(client_fd);
    }
}
```

#### 반환값

현재 최소 구현에서는 무한 루프이므로 정상 반환하지 않아도 된다.

실패로 loop를 종료하는 경우:

```c
-1
```

### `int http_read_request(int fd, HttpRequest *out_req, char *errbuf, size_t errbuf_size)`

#### 인자

```c
int fd
HttpRequest *out_req
char *errbuf
size_t errbuf_size
```

#### 동작

소켓에서 HTTP request를 읽고 파싱한다.

세부 흐름:

```text
1. out_req 초기화
2. recv()로 header를 읽는다
3. "\r\n\r\n"를 찾는다
4. header가 HTTP_MAX_HEADER_SIZE를 넘으면 실패
5. 첫 줄에서 method, path 추출
6. POST 요청이면 Content-Length 헤더 찾기
7. Content-Length 파싱
8. body_len이 HTTP_MAX_BODY_SIZE를 넘으면 실패
9. 이미 읽은 body 일부가 있으면 복사
10. 부족한 body는 recv()로 추가 읽기
11. body를 null-terminated string으로 저장
```

#### 반환값

```c
0   성공
-1  실패
```

#### 실패 시

`errbuf`에 사람이 읽을 수 있는 에러 메시지를 넣는다.

예:

```text
missing Content-Length
invalid Content-Length
request header too large
request body too large
malformed request line
```

### `int write_all(int fd, const char *buf, size_t len)`

#### 인자

```c
int fd
const char *buf
size_t len
```

#### 동작

`send()`가 일부만 전송하는 경우를 고려해 전체 buffer를 끝까지 전송한다.

Pseudo code:

```c
size_t sent = 0;

while (sent < len) {
    ssize_t n = send(fd, buf + sent, len - sent, 0);

    if (n < 0) {
        if (errno == EINTR) {
            continue;
        }
        return -1;
    }

    if (n == 0) {
        return -1;
    }

    sent += n;
}

return 0;
```

#### 반환값


# 10. `task_queue.h` / `task_queue.c` 명세

## 10.1 목적

Main Thread가 accept한 client socket fd를 Worker Thread에 전달하기 위한 bounded queue다.

---

## 10.2 상수

```c
#define TASK_QUEUE_CAPACITY 1024
```

---

## 10.3 구조체

### `Task`

```c
typedef struct {
    int client_fd;
} Task;
```

---

### `TaskQueue`

```c
typedef struct {
    Task items[TASK_QUEUE_CAPACITY];
    size_t head;
    size_t tail;
    size_t count;
    int shutdown;

    pthread_mutex_t mutex;
    pthread_cond_t not_empty;
    pthread_cond_t not_full;
} TaskQueue;
```

#### 필드 의미

| 필드          | 의미                          |
| ----------- | --------------------------- |
| `items`     | circular queue 저장소          |
| `head`      | pop 위치                      |
| `tail`      | push 위치                     |
| `count`     | 현재 task 수                   |
| `shutdown`  | 종료 플래그                      |
| `mutex`     | queue 보호 mutex              |
| `not_empty` | pop 대기용 condition variable  |
| `not_full`  | push 대기용 condition variable |

---

## 10.4 함수

### `int task_queue_init(TaskQueue *queue)`

#### 인자

```c
TaskQueue *queue
```

#### 동작

queue를 초기화한다.

초기값:

```text
head = 0
tail = 0
count = 0
shutdown = 0
```

`pthread_mutex_init`, `pthread_cond_init`을 호출한다.

#### 반환값

```c
0   성공
-1  실패
```

---

### `void task_queue_destroy(TaskQueue *queue)`

#### 인자

```c
TaskQueue *queue
```

#### 동작

mutex와 condition variable을 destroy한다.

#### 반환값

없음.

---

### `int task_queue_try_push(TaskQueue *queue, Task task)`

#### 인자

```c
TaskQueue *queue
Task task
```

#### 동작

queue에 task를 넣는다.

이 함수는 blocking하지 않는다.

처리 규칙:

```text
1. mutex lock
2. shutdown이면 실패
3. count == TASK_QUEUE_CAPACITY이면 full 반환
4. items[tail] = task
5. tail = (tail + 1) % TASK_QUEUE_CAPACITY
6. count++
7. not_empty signal
8. mutex unlock
```

#### 반환값

```c
0   성공
1   queue full
-1  shutdown 또는 오류
```

---

### `int task_queue_push(TaskQueue *queue, Task task)`

#### 인자

```c
TaskQueue *queue
Task task
```

#### 동작

queue에 task를 넣는다.

queue가 가득 차 있으면 `not_full`을 기다린다.

이번 서버의 accept loop에서는 `task_queue_try_push()`를 우선 사용한다.
이 blocking push는 테스트 또는 향후 확장용으로 제공한다.

#### 반환값

```c
0   성공
-1  shutdown 또는 오류
```

---

### `int task_queue_pop(TaskQueue *queue, Task *out_task)`

#### 인자

```c
TaskQueue *queue
Task *out_task
```

#### 동작

queue에서 task 하나를 꺼낸다.

queue가 비어 있으면 `not_empty`를 기다린다.

처리 규칙:

```text
1. mutex lock
2. count == 0 && !shutdown이면 not_empty wait
3. shutdown && count == 0이면 실패 반환
4. *out_task = items[head]
5. head = (head + 1) % TASK_QUEUE_CAPACITY
6. count--
7. not_full signal
8. mutex unlock
```

#### 반환값

```c
0   성공
-1  shutdown으로 인해 더 이상 task 없음
```

---

### `void task_queue_shutdown(TaskQueue *queue)`

#### 인자

```c
TaskQueue *queue
```

#### 동작

`shutdown = 1`로 설정하고 대기 중인 thread를 모두 깨운다.

호출할 signal:

```c
pthread_cond_broadcast(&queue->not_empty);
pthread_cond_broadcast(&queue->not_full);
```

#### 반환값

없음.

---

### `size_t task_queue_size(TaskQueue *queue)`

#### 인자

```c
TaskQueue *queue
```

#### 동작

현재 queue에 쌓인 task 개수를 반환한다.

mutex로 보호해서 읽는다.

#### 반환값

# 14. SELECT-only batch 처리 명세

## 14.1 왜 별도 처리가 필요한가

SELECT는 read lock으로 병렬 처리하고 싶다.

하지만 기존 DB 엔진은 테이블을 lazy loading한다.

SELECT라도 최초 테이블 접근 시 다음 작업이 발생할 수 있다.

```text
schema load
.data 파일 생성
B+Tree index rebuild
ExecutionContext.tables append
```

이 작업들은 read가 아니라 write 성격이다.

따라서 SELECT-only batch는 실제 SELECT 실행 전에 write lock으로 필요한 table runtime을 preload해야 한다.

---

## 14.2 함수

### `static int db_api_execute_read_batch(DbApi *api, TokenArray *tokens, char **out_json, int *out_http_status)`

#### 인자

```c
DbApi *api
TokenArray *tokens
char **out_json
int *out_http_status
```

#### 동작

SELECT-only SQL batch를 실행한다.

세부 흐름:

```text
1. StatementList list 생성
2. parse_all_statements(tokens, &list)
3. list가 비어 있으면 error
4. list 안에 INSERT가 있으면 내부 오류
5. SELECT table name 목록 수집
6. write lock 획득
7. 각 table runtime preload
8. write lock 해제
9. read lock 획득
10. list의 각 statement execute_statement()
11. ExecResult를 JSON results 배열에 append
12. read lock 해제
13. StatementList 해제
```

#### 반환값

```c
STATUS_OK
STATUS_PARSE_ERROR
STATUS_SCHEMA_ERROR
```

# 16. INSERT 포함 batch 처리 명세

## 16.1 원칙

SQL batch 안에 INSERT가 하나라도 있으면 전체 batch를 write lock으로 실행한다.

예:

```sql
INSERT INTO users VALUES ('kim', 25);
SELECT * FROM users WHERE id = 1;
```

위 요청은 write lock으로 전체 실행한다.

이유:

* INSERT가 `.data`, B+Tree, `next_id`를 수정한다.
* 같은 요청 안의 SELECT는 직전 INSERT 결과를 볼 수 있어야 한다.
* 중간 상태를 다른 worker가 보면 안 된다.

---

## 16.2 함수

### `static int db_api_execute_write_batch(DbApi *api, TokenArray *tokens, char **out_json, int *out_http_status)`

#### 인자

```c
DbApi *api
TokenArray *tokens
char **out_json
int *out_http_status
```

#### 동작

INSERT가 포함된 SQL batch를 write lock으로 실행한다.

세부 흐름:

```text
1. write lock 획득
2. token cursor = 0
3. JSON results array 시작
4. while EOF가 아닐 때:
     parse_next_statement(tokens, &cursor, &stmt)
     parse 실패 시:
        error JSON 생성
        break
     execute_statement(&api->ctx, stmt, &result)
     execute 실패 시:
        error JSON 생성
        free stmt
        break
     ExecResult를 JSON results에 append
     free_exec_result()
     free_statement()
5. write lock 해제
6. 최종 JSON 반환
```

#### 중요한 정책

이 함수는 기존 CLI와 비슷하게 동작한다.

즉:

```sql
INSERT INTO users VALUES ('kim', 25);
SELCT * FROM users;
```

첫 번째 INSERT가 이미 성공한 뒤 두 번째 statement parse error가 나면 rollback하지 않는다.

응답 JSON에는 성공한 결과와 error를 함께 담는다.

예:

```json
{
  "status": "error",
  "code": "PARSE_ERROR",
  "message": "unexpected token",
  "results": [
    {
      "type": "insert",
      "affected_rows": 1,
      "generated_id": 1
    }
  ]
}
```

#### 반환값

```c
STATUS_OK 또는 기존 StatusCode
```

# 23. 동시성 제어 명세

## 23.1 핵심 정책

무잠금 SELECT는 금지한다.

대신 `pthread_rwlock_t`를 사용한다.

```text
SELECT-only batch:
  write lock으로 table preload
  read lock으로 SELECT 실행

INSERT 포함 batch:
  write lock으로 전체 실행
```

---

## 23.2 SELECT-only 흐름

```text
1. SQL tokenize
2. 모든 statement parse
3. 모든 statement가 SELECT인지 확인
4. SELECT에서 사용하는 table 이름 수집
5. write lock 획득
6. table runtime preload
7. write lock 해제
8. read lock 획득
9. SELECT 실행
10. read lock 해제
```

---

## 23.3 INSERT 포함 흐름

```text
1. SQL tokenize
2. write lock 획득
3. statement 하나 parse
4. execute
5. 결과 JSON append
6. 다음 statement 반복
7. 에러 발생 시 partial results 포함해서 error JSON 생성
8. write lock 해제
```

---

## 23.4 왜 이렇게 하는가

* 여러 SELECT는 동시에 실행 가능해야 한다.
* INSERT는 `.data`, `next_id`, B+Tree를 수정하므로 단독 실행해야 한다.
* SELECT도 lazy loading 시 `ExecutionContext`를 수정할 수 있으므로 preload가 필요하다.
* B+Tree split 중 SELECT가 무잠금으로 접근하면 깨진 중간 상태를 읽을 수 있다.

# 25. 메모리 관리 규칙

# 30. Codex에게 보낼 최종 요약 지시문

아래 문단을 Codex 입력 맨 앞이나 맨 뒤에 붙이면 된다.

```text
기존 C 기반 sql_processor는 유지하고, mini_db_server라는 HTTP/JSON API 서버를 새로 추가하라.

mini_db_server는 ./mini_db_server -d <db_dir> -p <port> -t <threads> 형태로 실행된다.
Main Thread는 socket accept만 수행하고, client_fd를 Bounded Task Queue에 넣는다.
Worker Thread는 queue에서 client_fd를 꺼내 HTTP request parsing, JSON body parsing, SQL 실행, JSON response 전송, close를 담당한다.

API는 GET /health, POST /query, GET /stats를 구현한다.
POST /query는 {"sql":"..."} JSON body를 받고, 기존 lexer/parser/executor/storage/B+Tree 기반 DB 엔진으로 SQL을 실행한다.

동시성 제어는 pthread_rwlock_t를 사용한다.
무잠금 SELECT는 금지한다.
SELECT-only batch는 write lock으로 table runtime을 preload한 뒤 read lock으로 실행한다.
INSERT가 하나라도 포함된 batch는 전체를 write lock으로 실행한다.

Task Queue는 capacity 1024의 bounded circular queue로 구현한다.
queue가 가득 차면 503 JSON error를 반환한다.

ExecResult는 JSON으로 변환한다.
SELECT 결과에는 type, used_index, columns, rows, row_count를 포함한다.
INSERT 결과에는 type, affected_rows, generated_id를 포함한다.

외부 JSON 라이브러리는 사용하지 않는다.
json_extract_string_field(), JsonBuffer, write_all()을 직접 구현한다.

기존 make test는 계속 통과해야 하며, tests/test_api_server.sh를 추가해 /health, INSERT, SELECT, used_index, invalid SQL, explicit id insert error, concurrent SELECT, concurrent INSERT, /stats를 검증한다.
```
