## Step 2-3. `db_api.c` 역할 결정

기존 DB 엔진을 그대로 재사용하기 위해, 중간에 db_api.c라는 adapter 계층을 두었습니다.

db_api.c는 전차수 프로젝트의 lexer → parser → executor 흐름을 그대로 호출하면서
API 서버에 필요한 정책만 담당하는 역할을 합니다.

즉, 문법 검사는 parser가 담당하고,
db_api.c는 그 결과를 기반으로 요청을 제어하고 가공합니다.

구체적으로는 요청당 SQL이 1문장인지 확인하고,
SELECT인지 INSERT인지 구분한 뒤,
상황에 맞는 lock 정책을 적용합니다.

그리고 실행 결과인 ExecResult를 JSON으로 변환하고,
내부 에러를 HTTP 상태 코드로 매핑하여 최종 응답을 구성합니다.

이렇게 역할을 분리함으로써,
DB 엔진은 SQL 처리에 집중하고,
API 서버는 요청 처리와 응답 구성에 집중할 수 있도록 설계했습니다.

# Step 3. 아키텍처와 동시성 설계

## Step 3-1. 전체 아키텍처

전체 구조는 다음과 같습니다.

```text
Client
  ↓ HTTP/JSON
mini_db_server
  ↓ accept()
Bounded Queue
  ↓
Thread Pool
  ↓
db_api.c
  ↓
lexer / parser / executor
  ↓
storage + B+Tree
```

서버의 main thread는 SQL을 직접 처리하지 않습니다.

main thread는 `accept()`로 클라이언트 연결을 받은 뒤, 그 연결을 Bounded Queue에 넣습니다.

그 다음 Worker Thread가 Queue에서 작업을 꺼내 HTTP 요청을 읽고, JSON에서 SQL을 추출한 뒤, `db_api.c`를 통해 기존 SQL 엔진을 호출합니다.

### Mermaid 위치 추천

```text
전체 요청 처리 시퀀스 다이어그램
```

## Step 3-2. Bounded Queue 구조

작업 대기 큐는 무제한 큐가 아니라 Bounded Queue로 만들었습니다.

기본 queue capacity는 64로 시작했습니다.

무제한 큐를 사용하면 요청이 폭주할 때 메모리 사용량과 대기 시간이 계속 증가합니다.

최악의 경우 서버가 불안정해질 수 있습니다.

그래서 Queue에 상한선을 두고, Queue가 가득 차면 일부 요청을 거절하도록 설계했습니다.

이 방식은 서버가 무리해서 모든 요청을 쌓아두는 대신, 처리 가능한 범위 안에서 안정적으로 동작하도록 만드는 선택입니다.

### Mermaid 위치 추천

```text
Queue 정상 처리 / Queue Full 시 요청 거절 흐름
```

## Step 3-3. Worker Thread 개수 기준

기본 Worker Thread 수는 4개로 시작했습니다.

이 Work Thread는 네트워크 요청을 받고, `.data` 파일을 읽고 쓰는 I/O 작업을 주로 합니다.

그래서 Worker Thread 수는 CPU 코어 수와 정확히 같게 고정하기보다는, I/O 대기 시간을 고려해 더 크게 둘 수도 있습니다.

기본 값은 4로 시작하고 최적값을 찾기 위한 benchmark 테스트 기본 조합은 2, 4, 8로 두었습니다. 성능 검증 이후 최적값으로 세팅합니다.

즉, 현재 기본값은 최종 정답이 아니라 실험을 위한 안전한 시작값입니다.

## Step 3-4. SELECT 락 정책 고민

첫 번째는 SELECT를 항상 readlock으로 처리하는 방식입니다.
하지만 저희 시스템에서는 처음 SELECT 시 preload 과정이 발생하며, 이 과정에서 schema 로드, 인덱스 구성, 캐시 추가 등 내부 상태가 변경됩니다.
따라서 여러 스레드가 동시에 이 과정을 수행하면 충돌 가능성이 있어 무잠금 방식은 적절하지 않다고 판단했습니다.

두 번째는 SELECT까지 포함해 모든 요청을 write lock으로 처리하는 방식입니다.
이 경우 동시성 문제는 확실히 해결되지만, 모든 요청이 순차적으로 처리되어 성능상 이점이 사라지고 사실상 싱글 스레드처럼 동작하게 됩니다.

세 번째는 SELECT와 INSERT를 구분하는 방식입니다.
SELECT는 preload 단계에서만 write lock을 사용하고, 이후 실제 조회는 read lock으로 처리하며, INSERT는 전체를 write lock으로 보호합니다.

최종적으로 저희는 세 번째 방식을 선택했습니다.
즉, 초기 preload 과정은 write lock으로 보호하고, 이후 조회는 read lock으로 병렬 처리함으로써 데이터 정합성과 성능을 동시에 확보했습니다.



### Mermaid 위치 추천

```text
SELECT 처리 흐름: preload write lock → read lock 실행
```

## Step 3-5. INSERT 락 정책 고민

INSERT는 `.data` 파일 append, `next_id` 증가, B+Tree insert를 함께 수행합니다.

이 작업들이 동시에 여러 thread에서 실행되면 id가 중복되거나, 파일 append가 꼬이거나, B+Tree 구조가 깨질 수 있습니다.

그래서 INSERT는 전역 write lock으로 보호했습니다.

table-level lock이나 더 세밀한 lock도 가능하지만, 하루짜리 프로젝트에서는 구현 복잡도 대비 이득이 크지 않다고 판단했습니다.

전역 write lock은 단순하지만 안전하고, 이번 MVP 범위에는 가장 적합했습니다.

### Mermaid 위치 추천

```text
INSERT 처리 흐름: write lock → append / next_id / B+Tree insert → unlock
```

# Step 9. Postman 시연 5 — Concurrent SELECT

단일 SQL만 허용한다고 해서 동시성 검증을 못 하는 것은 아닙니다.

동시성은 한 요청 안에 SQL이 여러 개 있는지보다, **여러 요청이 동시에 들어오는지**에서 발생합니다.

그래서 저희는 Postman Performance Test로 단일 SQL 요청을 여러 개 동시에 보내는 방식으로 검증했습니다.

Concurrent SELECT 요청은 다음과 같이 준비합니다.

```text
Postman Collection: Concurrent SELECT

Request:
Method: POST
URL: http://127.0.0.1:8080/query

Headers:
Content-Type: application/json

Body → raw → JSON:
{
  "sql": "SELECT * FROM users WHERE id = 1;"
}
```

Performance Test 설정:

```text
Virtual Users: 20
Duration: 10 seconds
Load Profile: Fixed
```

Postman Performance Test는 Collection Runner에서 실행할 수 있고, duration과 virtual user 수를 설정할 수 있습니다. Fixed profile을 사용하면 설정한 virtual user 수가 테스트 동안 유지됩니다. ([Postman Docs][1])

이건 read-read 상황입니다.
여러 SELECT가 동시에 들어와도 preload 이후 read lock으로 안정적으로 병렬 처리되는지 확인합니다.

# Step 11. 시행착오와 배운 점

이제 구현 설명에서 한 단계 더 나아가서, 저희가 직접 실험하면서 어떤 시행착오를 겪었는지 말씀드리겠습니다.

처음에는 멀티스레딩 서버를 만들면 worker 수를 늘릴수록 당연히 더 빨라질 것이라고 생각했습니다.
그래서 thread pool, bounded queue, `pthread_rwlock_t`를 넣은 뒤, benchmark를 통해 성능이 좋아지는 모습을 보여주는 것을 목표로 했습니다.

### Mermaid 추천

```mermaid
flowchart LR
    H["가설<br/>멀티스레딩이면 무조건 더 빠르다"] --> E["실험<br/>worker, queue, workload benchmark"]
    E --> X["반례<br/>worker를 늘려도 항상 더 빠르지 않았다"]
    X --> A["원인 분석<br/>전역 rwlock + 짧은 작업 + 오버헤드"]
    A --> L["배운 점<br/>스레드 수보다 lock 범위가 더 중요하다"]
    L --> R["후속 실험<br/>lock 범위를 더 좁혀봄"]
    R --> I["관찰 결과<br/>멀티스레딩 이점이 더 잘 드러남"]
```

## Step 11-1. 처음 가설

가설은 단순했습니다.

멀티스레딩이면 여러 worker가 동시에 요청을 처리하니까, worker 수가 많을수록 throughput도 같이 올라갈 것이라고 생각했습니다.

## Step 11-2. 실제 반례

그런데 실제 benchmark 결과는 예상과 달랐습니다.

worker를 늘려도 항상 더 빨라지지 않았고, 어떤 조합에서는 오히려 worker가 적은 쪽이 더 빠르게 나오기도 했습니다.

즉, `멀티스레딩 = 무조건 속도 향상`이라는 가설이 바로 깨졌습니다.

## Step 11-3. 원인 분석

원인은 멀티스레딩 자체가 아니라, 실제로 병렬로 실행될 수 있는 구간이 얼마나 남아 있느냐였습니다.

현재 기본 구현은 correctness를 우선한 전역 `pthread_rwlock_t` 기반입니다.

`INSERT`는 write lock으로 직렬화되고, `SELECT`도 preload 단계에서는 write lock을 거친 뒤에야 read lock으로 실행됩니다.

또 우리 workload의 핵심 쿼리는 `WHERE id = ?` 형태라 작업이 매우 짧아서, SQL 실행 시간보다 queue, lock, context switch 오버헤드가 더 크게 보이는 경우가 있었습니다.

즉, worker thread는 여러 개였지만 DB 안쪽의 중요한 구간은 충분히 병렬화되지 못했고, 그래서 기대한 만큼 속도 향상이 바로 나타나지 않았습니다.

## Step 11-4. 여기서 배운 점

이 실험을 통해 저희가 얻은 핵심 교훈은 세 가지입니다.

첫째, 멀티스레딩은 무조건 더 빠른 기술이 아닙니다.

둘째, 속도는 thread 수보다 `lock granularity`, 공유 상태, 작업 길이에 더 크게 영향을 받습니다.

셋째, 서버에서 멀티스레딩의 가치는 단일 요청 1개를 극단적으로 빠르게 끝내는 것보다, 여러 요청이 동시에 들어와도 시스템이 버티게 하는 데 더 가깝습니다.

## Step 11-5. 후속 실험

그래서 이후에는 전역 lock이 병목이라면, lock 범위를 더 좁히면 멀티스레딩 이점이 살아나는지 별도 실험도 해봤습니다.

그 경우 서로 덜 충돌하는 요청이 동시에 통과할 수 있어서, worker 수가 많은 구성이 더 유리한 결과도 확인했습니다.

다만 현재 repo의 기본 구현은 여전히 전역 `rwlock` 기반 MVP입니다.

정리하면 저희 프로젝트는 기존 CLI 기반 SQL Processor를 HTTP/JSON API 서버로 확장한 작업입니다.

설계상 중요한 결정은 네 가지입니다.

첫째, raw TCP가 아니라 HTTP/JSON을 선택해서 테스트와 발표, 구현 난이도의 균형을 맞췄습니다.

둘째, 요청당 SQL 1문장만 허용해서 batch 처리의 복잡도를 줄이고, 병렬 처리와 동시성 제어에 집중했습니다.

셋째, 기존 lexer, parser, executor, runtime, storage, B+Tree를 그대로 재사용하고, `db_api.c`는 API 정책과 락 제어를 담당하는 adapter로 분리했습니다.

넷째, `pthread_rwlock_t`를 사용해서 SELECT는 preload 후 read lock, INSERT는 write lock으로 처리했습니다.
이를 통해 SELECT 병렬성과 INSERT 정합성을 동시에 확보했습니다.

앞으로 개선한다면 SQL batch, `/stats`, table-level lock, 더 정교한 benchmark 기반 thread/queue 튜닝, UPDATE/DELETE까지 확장할 수 있습니다.

감사합니다.

# QnA 대비 답변

## Q. 왜 TCP가 아니라 HTTP/JSON인가요?

raw TCP는 직접 요청 구분 방식과 응답 포맷을 설계해야 해서 구현과 설명 비용이 큽니다.
이번 프로젝트는 하루 안에 API 서버, Thread Pool, DB 엔진 연동, 동시성 제어를 보여주는 것이 핵심이었기 때문에, **Postman으로 바로 테스트 가능하고 요청/응답 구조가 명확한 HTTP/JSON**을 선택했습니다.

---

## Q. 단일 SQL만 받으면 동시성 검증이 약하지 않나요?

아닙니다.
동시성은 한 요청 안의 SQL 개수가 아니라 여러 요청이 동시에 들어올 때 발생합니다.
저희는 Postman Performance Test로 단일 SELECT 요청 여러 개, 단일 INSERT 요청 여러 개, 그리고 SELECT/INSERT mixed workload를 동시에 보내서 read-read, write-write, read-write 상황을 검증했습니다.

---

## Q. 왜 SELECT를 완전 무잠금으로 두지 않았나요?

기존 엔진은 첫 테이블 접근 때 `get_or_load_table_runtime()`에서 schema load, `.data` 확인, B+Tree rebuild, runtime cache append를 수행할 수 있습니다.
즉 첫 SELECT도 shared state를 수정할 수 있기 때문에 완전 무잠금 SELECT는 위험합니다.
그래서 SELECT는 먼저 write lock으로 preload하고, 실제 조회는 read lock으로 실행했습니다.

---

## Q. 왜 mutex 하나가 아니라 rwlock인가요?

mutex 하나로 전체 DB를 보호하면 구현은 쉽지만 SELECT도 모두 직렬화되어 DB 레벨에서는 싱글 스레드처럼 동작합니다.
rwlock을 쓰면 여러 SELECT는 preload 이후 read lock 구간에서 함께 처리할 수 있고, INSERT만 단독 처리할 수 있어서 병렬성과 정합성을 모두 설명할 수 있습니다.

---

## Q. 왜 queue에 상한을 뒀나요?

상한이 없는 queue는 요청 폭주 시 메모리 사용량과 대기 시간이 계속 증가합니다.
최악의 경우 서버가 불안정해질 수 있습니다.
그래서 Bounded Queue를 두고, queue가 꽉 차면 503으로 거절하는 방식으로 시스템 안정성을 확보했습니다.
