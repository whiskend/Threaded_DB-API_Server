# Step 1. 발표 시작

안녕하세요. 7조 발표 시작하겠습니다.

저희 팀은 기존 DB 엔진을 재사용하면서, 그 앞에 `mini_db_server`라는 HTTP API 서버를 추가했습니다.

---

# Step 2. 주요 설계 의사결정

## Step 2-1. 통신 방식 결정

우선 통신 방식인 raw TCP와 HTTP/JSON 사이에서 고민했습니다.

raw TCP는 유연하지만, 프로토콜 설계와 테스트 환경까지 직접 구성해야 해서
일일 프로젝트에는 부담이 커 HTTP/JSON을 선택했습니다.

HTTP/JSON방식은 Postman이라는 API 테스트 도구를 이용해
바로 요청을 보내고 결과를 확인할 수 있고, 더불어 동시에 여러 요청을 보내 동시성 상황도 쉽게 검증할 수 있습니다. 또한 프로토콜 구조를 시각적으로 보여주기 쉬워 API 서버라는 과제 목적에도 더 적합합니다.

---

## Step 2-2. 요청당 SQL 개수 결정

두 번째로 고민한 부분은 한 요청에 여러 SQL문장을 보내는 것을 허용할지, 한 문장만 받을지였습니다.

한 요청에 여러 SQL을 담는 batch 방식은 응답 형식이 복잡해지고 트랜섹션 문제가 발생합니다.

그래서 이번 프로젝트에서는 **요청당 SQL 1문장만 허용**했습니다.


---

## Step 2-3. `db_api.c` 역할 결정

세 번째로 기존 DB 엔진을 그대로 재사용하기 위해, 중간에 db_api.c라는 adapter 계층을 두었습니다.

db_api.c는 전차수 프로젝트의 lexer → parser → executor을 그대로 호출하면서
API 서버의 정책을 적용하고, 실행 결과 및 내부 에러를 HTTP 형식으로 매핑하여 최종 응답을 구성하는 역할을 합니다.

이렇게 역할을 분리함으로써,
DB 엔진은 SQL 처리에 집중하고,
API 서버는 요청 처리와 응답 구성에 집중할 수 있습니다.

—

# Step 3. 아키텍처와 동시성 설계

## Step 3-1. 전체 아키텍처

main thread는 `accept()`로 클라이언트 연결을 받은 뒤, 그 연결을 Bounded Queue에 넣습니다.

그 다음 Worker Thread가 Queue에서 작업을 꺼내 HTTP 요청을 읽고, JSON에서 SQL을 추출한 뒤, `db_api.c`를 통해 기존 SQL 엔진을 호출합니다.


---


## Step 3-2. Bounded Queue 구조

작업 대기 큐는 사이즈 제한이 있는 Bounded Queue로 만들었습니다.

무제한 큐를 사용하면 요청이 폭주할 때 메모리 사용량과 대기 시간이 계속 증가합니다.

최악의 경우 서버가 불안정해질 수 있습니다.

그래서 Queue에 상한선을 두고, Queue가 가득 차면 일부 요청을 거절하도록 설계했습니다.

이 방식은 서버가 무리해서 모든 요청을 쌓아두는 대신, 처리 가능한 범위 안에서 안정적으로 동작하도록 만드는 선택입니다.

최적값을 찾기 위해선 실험이 필요했기에 다양한 값으로 benchmark 테스트를 하고 최적값으로 세팅합니다.
benchmark 테스트 값으로 32, 64, 128을 두었습니다.


---

## Step 3-3. Worker Thread 개수 기준

이 Work Thread는 네트워크 요청을 받고, `.data` 파일을 읽고 쓰는 I/O 작업을 주로 합니다.

그래서 Worker Thread 수는 CPU 코어 수와 정확히 같게 고정하기보다는, I/O 대기 시간을 고려해 더 크게 둘 수도 있습니다.

큐 사이즈와 마찬가지로 최적값을 찾기 위해선 실험이 필요했기에 다양한 값으로 benchmark 테스트를 하고 최적값으로 세팅합니다.
benchmark 테스트 값으로 2, 4, 8을 두었습니다.

---

## Step 3-4. INSERT 락 정책 고민

INSERT는 전역 write lock으로 보호했습니다. 더 세밀한 lock도 가능하지만, 일일 프로젝트에서는 구현 복잡도 대비 이득이 크지 않기에 전역 write lock이 이번 MVP 범위에는 가장 적합하다 판단했습니다만, 추후 생각치 못한 부분이 발견되었습니다. 해당 부분은 벤치마크 파트에서 말씀드리겠습니다.

SELECT의 경우, preload 단계에서만 write lock을 사용하고, 이후 실제 조회는 read lock으로 처리합니다.
항상 readlock으로 처리하거나 write lock으로 처리하면 여러 스레드가 동시에 이 과정을 수행하면 충돌 가능성이 있거나, 모든 요청이 순차적으로 처리되어 사실상 싱글 스레드처럼 동작하게 되어 성능상 이점이 사라집니다.

# Step 11. 시행착오와 배운 점

마지막으로 저희가 직접 실험하면서 어떤 시행착오를 겪었는지 말씀드리겠습니다.

처음에는 멀티스레딩 서버를 만들면 worker 수를 늘릴수록 당연히 더 빨라질 것이라고 생각했습니다.


## Step 11-1. 처음 가설

가설은 단순했습니다.

멀티스레딩이면 여러 worker가 동시에 요청을 처리하니까, worker 수가 많을수록 처리량도 같이 올라갈 것이라고 생각했습니다.

## Step 11-2. 실제 반례

그런데 실제 benchmark 결과는 예상과 달랐습니다.

worker를 늘려도 항상 더 빨라지지 않았고, 대부분의 조합에서는 오히려 worker가 적은 쪽이 더 빠르게 나왔습니다.

## Step 11-3. 원인 분석


`INSERT`는 write lock, `SELECT` 단계에서는 write lock을 거친 뒤에야 read lock으로 실행됩니다.

때문에 SQL 실행 시간보다 queue, lock, context switch 오버헤드가 더 크게 보이는 경우가 있었습니다.

그래서 기대한 만큼 속도 향상이 바로 나타나지 않았습니다.


## Step 11-4. 여기서 배운 점


첫째, 멀티스레딩은 무조건 더 빠른 기술이 아닙니다.

둘째, 속도는 thread 수보다 공유 상태, 작업 길이에 더 크게 영향을 받습니다.

셋째, 서버에서 멀티스레딩의 가치는 단일 요청 1개를 극단적으로 빠르게 끝내는 것보다, 여러 요청이 동시에 들어와도 시스템이 버티게 하는 데 더 가깝습니다.


## Step 11-5. 후속 실험


전역 lock이 병목이라면, lock 범위를 더 좁히면 멀티스레딩 이점이 살아나는지 별도 실험도 해봤습니다.

그 결과 속도는 빨라졌습니다. 하지만 그걸로 인해서 발생하는 B+트리 붕괴에 대해서는 추가로 공부해야할 부분입니다.

다만 현재 repo의 기본 구현은 여전히 전역 `rwlock` 기반 MVP입니다.
