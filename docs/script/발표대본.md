# Step 2. 주요 설계 의사결정

## Step 2-1. 통신 방식 결정

저희가 처음에 가장 많이 고민한 부분은 통신 방식을 raw TCP로 할지, HTTP/JSON으로 할지였습니다.

raw TCP를 쓰면 직접 프로토콜을 설계할 수 있다는 장점은 있습니다.

하지만 요청을 어디서 끊을지, 응답 포맷을 어떻게 만들지, 테스트 클라이언트를 어떻게 구성할지까지 직접 정해야 합니다.

하루짜리 프로젝트 범위에서는 이 부분이 리스크가 크다고 판단했습니다.

그래서 최종적으로는 **HTTP/JSON**으로 결정했습니다.

이 방식은 Postman으로 바로 테스트할 수 있고, 발표 때도 Method, URL, Header, Body, Response 구조를 화면으로 보여주기 쉽습니다.

또한 HTTP 상태 코드도 활용할 수 있어서 API 서버라는 과제 목적에 더 잘 맞았습니다.

```text

```

## Step 2-2. 요청당 SQL 개수 결정

두 번째로 고민한 부분은 한 요청에 SQL batch를 허용할지, SQL 1문장만 받을지였습니다.

한 요청에 여러 SQL을 담는 batch 방식도 가능하지만, 그렇게 하면 락 범위, 부분 실패 처리, 응답 형식이 한꺼번에 복잡해집니다.

예를 들어 첫 번째 INSERT는 성공했는데 두 번째 SELECT에서 실패하면, 어디까지 성공으로 볼지, 응답을 어떻게 표현할지, rollback을 할지 같은 문제가 생깁니다.

그래서 이번 프로젝트에서는 **요청당 SQL 1문장만 허용**했습니다.

대신 여러 요청을 동시에 보내는 방식으로 병렬 처리와 동시성 제어를 검증했습니다.

즉, 동시성 검증은 한 요청 안의 SQL 개수가 아니라, 여러 HTTP 요청이 동시에 들어오는 상황으로 확인했습니다.

### Mermaid 위치 추천

```text
Batch SQL 복잡도 vs 단일 SQL 요청 구조 비교
```

## Step 2-3. 기존 엔진 재사용 결정

세 번째 결정은 서버용 SQL 실행기를 새로 만들지, 기존 `sql_processor` 엔진을 재사용할지였습니다.

저희는 기존의 `lexer`, `parser`, `executor`, `runtime`, `storage`, `B+Tree`를 그대로 재사용하기로 했습니다.

이유는 이미 검증된 SQL 처리 흐름이 있는데, 서버용 실행기를 새로 만들면 구현 범위와 리스크가 모두 커지기 때문입니다.

이번 과제의 핵심은 새 DB 엔진 구현이 아니라, 기존 엔진을 API 서버로 확장하는 것입니다.

## Step 2-4. `db_api.c` 역할 결정

기존 엔진을 그대로 재사용하기 위해, 중간에 `db_api.c`라는 adapter 계층을 두었습니다.

`db_api.c`는 SQL 문법을 새로 검사하는 파일이 아닙니다.

SQL 문법 검사는 기존 parser가 담당합니다.

`db_api.c`는 기존 `lexer → parser → executor` 흐름을 호출하면서, API 서버에 필요한 정책을 담당합니다.

주요 역할은 다음과 같습니다.

```text
1. 요청당 SQL 1문장인지 확인
2. SELECT인지 INSERT인지 판단
3. lock 정책 결정
4. ExecResult를 JSON으로 변환
5. HTTP 에러 코드로 매핑
```

이렇게 나누면 기존 DB 엔진과 API 서버의 책임이 분리됩니다.

### Mermaid 위치 추천

```text
server.c → db_api.c → lexer/parser/executor 흐름도
```

## Step 2-5. AST 재사용 결정

`db_api.c`에서 SQL을 한 번 파싱하고, 뒤에서 다시 파싱하는 구조가 될 수도 있어서 이 부분도 고민했습니다.

최종적으로는 AST를 한 번만 만들고, 그 같은 AST를 정책 판단과 실행에 같이 사용했습니다.

즉, parser가 만든 `Statement *`를 보고 SELECT인지 INSERT인지 판단하고, 그 `Statement *`를 그대로 `execute_statement()`에 넘깁니다.

이렇게 하면 중복 파싱이 없고, 기존 executor 구조와도 자연스럽게 연결됩니다.

### Mermaid 위치 추천

```text
SQL → Token → AST → 정책 판단 + 실행 재사용 흐름
```

# Step 3. 아키텍처와 동시성 설계

## Step 3-1. 전체 아키텍처

전체 구조는 다음과 같습니다.

```text
Client
  ↓ HTTP/JSON
mini_db_server
  ↓ accept()
Bounded Queue
  ↓
Thread Pool
  ↓
db_api.c
  ↓
lexer / parser / executor
  ↓
storage + B+Tree
```

서버의 main thread는 SQL을 직접 처리하지 않습니다.

main thread는 `accept()`로 클라이언트 연결을 받은 뒤, 그 연결을 Bounded Queue에 넣습니다.

그 다음 Worker Thread가 Queue에서 작업을 꺼내 HTTP 요청을 읽고, JSON에서 SQL을 추출한 뒤, `db_api.c`를 통해 기존 SQL 엔진을 호출합니다.

```text

```

## Step 3-3. Bounded Queue 구조

작업 대기 큐는 무제한 큐가 아니라 Bounded Queue로 만들었습니다.

기본 queue capacity는 64로 시작했습니다.

무제한 큐를 사용하면 요청이 폭주할 때 메모리 사용량과 대기 시간이 계속 증가합니다.

최악의 경우 서버가 불안정해질 수 있습니다.

그래서 Queue에 상한선을 두고, Queue가 가득 차면 일부 요청을 거절하도록 설계했습니다.

이 방식은 서버가 무리해서 모든 요청을 쌓아두는 대신, 처리 가능한 범위 안에서 안정적으로 동작하도록 만드는 선택입니다.

### Mermaid 위치 추천

```text

```

## Step 3-4. Worker Thread 개수 기준

기본 Worker Thread 수는 4개로 시작했습니다.

이 서버는 순수 CPU-bound 작업만 하는 서버가 아닙니다.

네트워크 요청을 받고, `.data` 파일을 읽고 쓰는 I/O 작업도 포함합니다.

그래서 Worker Thread 수는 CPU 코어 수와 정확히 같게 고정하기보다는, I/O 대기 시간을 고려해 더 크게 둘 수도 있습니다.

다만 처음부터 근거 없이 큰 값을 고정하기보다, 4로 시작하고 8, 16, 32, 64 같은 값을 benchmark로 비교할 수 있게 했습니다.

즉, 현재 기본값은 최종 정답이 아니라 실험을 위한 안전한 시작값입니다.

## Step 3-5. SELECT 락 정책 고민

동시성 제어에서는 세 가지 선택지를 고민했습니다.

첫 번째는 SELECT를 완전히 무잠금으로 두는 것입니다.

두 번째는 SELECT까지 전부 write lock으로 막는 것입니다.

세 번째는 SELECT는 preload 후 read lock, INSERT는 write lock으로 나누는 방식입니다.

최종적으로는 세 번째 방식을 선택했습니다.

이유는 기존 `get_or_load_table_runtime()`이 첫 접근 때 schema load, `.data` 파일 확인, B+Tree 재구성, runtime cache append를 수행하기 때문입니다.

즉 첫 SELECT도 내부적으로 shared state를 수정할 수 있어서 완전 무잠금 SELECT는 위험합니다.

하지만 SELECT까지 전부 write lock으로 막으면, DB 레벨에서는 사실상 싱글 스레드처럼 동작합니다.

그래서 SELECT는 먼저 write lock으로 runtime을 preload한 뒤, 실제 조회는 read lock으로 실행하도록 했습니다.

### Mermaid 위치 추천

```text
SELECT 처리 흐름: preload write lock → read lock 실행
```

## Step 3-6. INSERT 락 정책 고민

INSERT는 `.data` 파일 append, `next_id` 증가, B+Tree insert를 함께 수행합니다.

이 작업들이 동시에 여러 thread에서 실행되면 id가 중복되거나, 파일 append가 꼬이거나, B+Tree 구조가 깨질 수 있습니다.

그래서 INSERT는 전역 write lock으로 보호했습니다.

table-level lock이나 더 세밀한 lock도 가능하지만, 하루짜리 프로젝트에서는 구현 복잡도 대비 이득이 크지 않다고 판단했습니다.

전역 write lock은 단순하지만 안전하고, 이번 MVP 범위에는 가장 적합했습니다.

### Mermaid 위치 추천

```text
INSERT 처리 흐름: write lock → append / next_id / B+Tree insert → unlock
```

## Step 3-7. `pthread_rwlock_t` 선택 이유

락은 단순 mutex 하나가 아니라 `pthread_rwlock_t`를 사용했습니다.

mutex 하나로 전체 DB를 보호하면 구현은 쉽지만, SELECT도 모두 직렬화됩니다.

그러면 DB 레벨에서는 사실상 싱글 스레드처럼 동작합니다.

반면 `pthread_rwlock_t`를 사용하면 여러 SELECT는 read lock으로 동시에 실행할 수 있고, INSERT는 write lock으로 단독 실행할 수 있습니다.

즉, `pthread_rwlock_t`는 병렬성과 데이터 정합성의 균형을 잡기 위한 선택입니다.

### Mermaid 위치 추천


## Step 3-8. 동시성 검증 방식

저희 서버는 요청당 SQL 1문장만 허용합니다.

하지만 동시성 검증은 가능합니다.

동시성은 한 요청 안에 SQL이 여러 개 있는지보다, 여러 요청이 동시에 들어오는지에서 발생하기 때문입니다.

그래서 저희는 단일 SQL 요청을 여러 개 동시에 보내서 다음 상황을 검증했습니다.

```text
1. concurrent SELECT
   → read-read 상황

2. concurrent INSERT
   → write-write 상황

3. SELECT + INSERT mixed workload
   → read-write 상황
```

이 방식으로 요청 단위는 단순하게 유지하면서도, 서버의 병렬 처리와 락 정책은 충분히 검증할 수 있습니다.

### Mermaid 위치 추천

```text

```

# Step 4. 서버 실행 시연

터미널에서 서버를 실행합니다.

```bash
./mini_db_server -d db -p 8080 -t 4 -q 64
```

저희 서버는 `-d`로 DB 디렉터리, `-p`로 포트, `-t`로 worker thread 수, `-q`로 queue capacity를 받습니다.

기본값은 worker 4개, queue 64로 시작했습니다.

worker 수는 I/O-bound 서버라는 점을 고려했습니다.
이 서버는 네트워크 요청과 파일 I/O가 있기 때문에 CPU 코어 수와 정확히 같게 두기보다는, 상황에 따라 코어 수의 2~3배까지도 실험해볼 수 있습니다.

다만 처음부터 큰 값을 고정하기보다, 4로 시작하고 8, 16, 32, 64를 benchmark로 비교할 수 있게 했습니다.

Queue도 마찬가지로 64에서 시작하고, 128, 256, 512, 1024 같은 값을 실험할 수 있게 했습니다.

Queue에 상한을 둔 이유는 요청이 폭주할 때 무제한으로 요청을 쌓으면 메모리 사용량과 응답 지연이 계속 증가하고, 최악의 경우 서버가 불안정해질 수 있기 때문입니다.
그래서 Bounded Queue를 두고, 가득 차면 일부 요청을 거절하는 방식으로 서버 안정성을 우선했습니다.

# Step 9. Postman 시연 5 — Concurrent SELECT

단일 SQL만 허용한다고 해서 동시성 검증을 못 하는 것은 아닙니다.

동시성은 한 요청 안에 SQL이 여러 개 있는지보다, **여러 요청이 동시에 들어오는지**에서 발생합니다.

그래서 저희는 Postman Performance Test로 단일 SQL 요청을 여러 개 동시에 보내는 방식으로 검증했습니다.

Concurrent SELECT 요청은 다음과 같이 준비합니다.

```text
Postman Collection: Concurrent SELECT

Request:
Method: POST
URL: http://127.0.0.1:8080/query

Headers:
Content-Type: application/json

Body → raw → JSON:
{
  "sql": "SELECT * FROM users WHERE id = 1;"
}
```

Performance Test 설정:

```text
Virtual Users: 20
Duration: 10 seconds
Load Profile: Fixed
```

Postman Performance Test는 Collection Runner에서 실행할 수 있고, duration과 virtual user 수를 설정할 수 있습니다. Fixed profile을 사용하면 설정한 virtual user 수가 테스트 동안 유지됩니다. ([Postman Docs][1])

이건 read-read 상황입니다.
여러 SELECT가 동시에 들어와도 preload 이후 read lock으로 안정적으로 병렬 처리되는지 확인합니다.

```text
Duration: 10 seconds
Load Profile: Fixed
```

이건 write-write 상황입니다.
여러 INSERT가 동시에 들어와도 write lock 덕분에 `.data` append, `next_id`, B+Tree insert가 깨지지 않는지 확인합니다.

추가로 SELECT와 INSERT를 섞은 mixed workload도 benchmark 대상으로 두었습니다.
실제 기본 thread 수와 queue 크기는 이런 workload를 기준으로 조정할 수 있게 했습니다.

Postman의 Performance Test 결과 화면에서는 virtual users, requests per second, average response time, error rate 같은 지표를 실시간으로 확인할 수 있습니다. ([Postman Docs][2])

# Step 11. 마무리

정리하면 저희 프로젝트는 기존 CLI 기반 SQL Processor를 HTTP/JSON API 서버로 확장한 작업입니다.

설계상 중요한 결정은 네 가지입니다.

첫째, raw TCP가 아니라 HTTP/JSON을 선택해서 테스트와 발표, 구현 난이도의 균형을 맞췄습니다.

둘째, 요청당 SQL 1문장만 허용해서 batch 처리의 복잡도를 줄이고, 병렬 처리와 동시성 제어에 집중했습니다.

셋째, 기존 lexer, parser, executor, runtime, storage, B+Tree를 그대로 재사용하고, `db_api.c`는 API 정책과 락 제어를 담당하는 adapter로 분리했습니다.

넷째, `pthread_rwlock_t`를 사용해서 SELECT는 preload 후 read lock, INSERT는 write lock으로 처리했습니다.
이를 통해 SELECT 병렬성과 INSERT 정합성을 동시에 확보했습니다.

앞으로 개선한다면 SQL batch, `/stats`, table-level lock, 더 정교한 benchmark 기반 thread/queue 튜닝, UPDATE/DELETE까지 확장할 수 있습니다.

# QnA 대비 답변

## Q. 왜 TCP가 아니라 HTTP/JSON인가요?

raw TCP는 직접 요청 구분 방식과 응답 포맷을 설계해야 해서 구현과 설명 비용이 큽니다.
이번 프로젝트는 하루 안에 API 서버, Thread Pool, DB 엔진 연동, 동시성 제어를 보여주는 것이 핵심이었기 때문에, **Postman으로 바로 테스트 가능하고 요청/응답 구조가 명확한 HTTP/JSON**을 선택했습니다.

---

## Q. 단일 SQL만 받으면 동시성 검증이 약하지 않나요?

아닙니다.
동시성은 한 요청 안의 SQL 개수가 아니라 여러 요청이 동시에 들어올 때 발생합니다.
저희는 Postman Performance Test로 단일 SELECT 요청 여러 개, 단일 INSERT 요청 여러 개, 그리고 SELECT/INSERT mixed workload를 동시에 보내서 read-read, write-write, read-write 상황을 검증했습니다.

---

## Q. 왜 SELECT를 완전 무잠금으로 두지 않았나요?

기존 엔진은 첫 테이블 접근 때 `get_or_load_table_runtime()`에서 schema load, `.data` 확인, B+Tree rebuild, runtime cache append를 수행할 수 있습니다.
즉 첫 SELECT도 shared state를 수정할 수 있기 때문에 완전 무잠금 SELECT는 위험합니다.
그래서 SELECT는 먼저 write lock으로 preload하고, 실제 조회는 read lock으로 실행했습니다.

---

## Q. 왜 mutex 하나가 아니라 rwlock인가요?

mutex 하나로 전체 DB를 보호하면 구현은 쉽지만 SELECT도 모두 직렬화되어 DB 레벨에서는 싱글 스레드처럼 동작합니다.
rwlock을 쓰면 여러 SELECT는 동시에 처리하고, INSERT만 단독 처리할 수 있어서 병렬성과 정합성을 모두 설명할 수 있습니다.

---

## Q. 왜 queue에 상한을 뒀나요?

상한이 없는 queue는 요청 폭주 시 메모리 사용량과 대기 시간이 계속 증가합니다.
최악의 경우 서버가 불안정해질 수 있습니다.
그래서 Bounded Queue를 두고, queue가 꽉 차면 503으로 거절하는 방식으로 시스템 안정성을 확보했습니다.
