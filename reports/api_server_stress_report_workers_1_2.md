# mini_db_server 고동시성 스트레스 리포트

## 처음 보는 분을 위한 안내

- 이 문서는 짧은 순간에 요청이 한꺼번에 몰릴 때 서버가 얼마나 잘 버티는지 보는 보고서입니다.
- 가장 먼저 볼 곳은 `버스트 수준별 최적 조합` 표입니다. 여기에는 갑작스러운 몰림 상황에서 가장 안정적이었던 설정만 추려져 있습니다.
- `워커 수`는 동시에 실제 일을 처리하는 인원 수, `큐 크기`는 잠시 대기시켜 둘 수 있는 줄 길이라고 생각하면 이해하기 쉽습니다.
- 좋은 조합은 보통 `성공`이 높고, `503`과 `오류`가 낮고, `p95 ms`와 `표준편차`가 작은 조합입니다.

## 용어 설명

- `성공`: 요청이 정상적으로 처리된 횟수입니다. 높을수록 좋습니다.
- `503`: 서버가 너무 바빠서 이번 요청은 받지 못하겠다고 돌려준 횟수입니다. 큐가 꽉 찼을 때 주로 나옵니다.
- `오류`: 정상 응답(JSON) 자체를 받지 못한 횟수입니다. 연결 실패나 클라이언트/OS 한계까지 포함될 수 있습니다.
- `평균 ms`: 전체 요청을 평균적으로 얼마나 빨리 처리했는지 보여줍니다.
- `p95 ms`: 느린 요청들까지 포함했을 때도 얼마나 괜찮은지 보여줍니다. 체감 품질을 볼 때 평균보다 더 중요합니다.
- `표준편차`: 같은 실험을 여러 번 반복했을 때 결과가 얼마나 흔들렸는지 나타냅니다. 작을수록 더 안정적입니다.

## 표 읽는 법

- `성공`이 높아도 `p95 ms`와 `표준편차`가 크면 실제 운영에서는 가끔씩 느려질 수 있습니다.
- 그래서 이 보고서는 `실패율 -> p95 지연 -> 표준편차 -> 더 작은 자원 사용` 순서로 조합을 비교합니다.

## 해석

- 이 스윕은 버스트 동시성만 따로 분리해서 봅니다. 일반 벤치 리포트와 달리 각 run의 요청 수는 동시 클라이언트 수와 같습니다.
- 각 조합을 10회 반복 측정했고, 성공/실패 수와 지연 시간 모두 평균과 표준편차를 같이 표시합니다.
- `혼합 (mixed)` 부하는 read lock 경로와 write lock 경로를 동시에 타기 때문에 우선적으로 해석해야 하는 지표입니다.
- `오류`가 0이 아니면, 포화 상태에서 클라이언트가 JSON 응답 자체를 받지 못하고 실패한 경우가 있었다는 뜻입니다. 이는 `503`만 세는 것보다 더 엄격한 기준입니다.
