안녕하세요. 7조 발표 시작하겠습니다.

저희 팀은 기존 DB 엔진을 재사용하면서, 그 앞에 `mini_db_server`라는 HTTP API 서버를 추가했습니다.

# Step 2. 주요 설계 의사결정

구현 전에 세 가지를 먼저 결정했습니다.

첫째, 통신 방식은 raw TCP 대신 HTTP/JSON을 선택했습니다.
raw TCP는 유연하지만 프로토콜과 테스트 환경까지 직접 만들어야 해서,
하루짜리 프로젝트에는 부담이 크다고 판단했습니다.
반면 HTTP/JSON은 Postman으로 외부클라이언트를 설정해 바로 테스트하고 시연하기 쉬웠습니다.

둘째, 요청당 SQL은 1문장만 허용했습니다.
여러 SQL을 한 번에 받는 batch 방식은 응답 형식과 실패 처리, 트랜잭션 문제를 함께 복잡하게 만들기 때문입니다.

셋째, 기존 DB 엔진을 그대로 재사용하기 위해 db_api라는 adapter 계층을 두었습니다.
db_api는 기존 lexer, parser, executor를 호출하고,
API 서버의 정책 적용과 응답 구성만 담당하도록 역할을 분리했습니다.

# Step 3. 아키텍처와 동시성 설계
## Step 3-1. 전체 아키텍처
main thread는 `accept()`로 받은 연결을 Bounded Queue에 넣고,
Worker Thread는 이를 꺼내 HTTP 요청을 처리한 뒤 `db_api.c`를 통해 기존 SQL 엔진을 호출합니다.
---
## Step 3-2. Bounded Queue 구조
**(내리기)** 작업 대기 큐는 요청 폭주 상황에 대비해 크기 제한이 있는 Bounded Queue로 설계했고,
큐 크기는 benchmark로 32, 64, 128을 비교해 결정했습니다.
Worker Thread는 I/O 대기 시간을 고려해 CPU 코어 수보다 더 크게 둘 수 있다고 보고,
thread 수 역시 benchmark로 2, 4, 8을 비교해 결정했습니다.

## Step 3-4. INSERT 락 정책 고민

**(축소하기)**
INSERT는 단순하고 안전한 전역 write lock으로 보호했습니다.
더 세밀한 lock도 가능하지만, 일일 프로젝트에서는 구현 복잡도 대비 이득이 크지 않다고 판단했습니다. 다만 이 선택의 한계는 뒤 벤치마크 파트에서 말씀드리겠습니다.
SELECT는 preload 단계만 write lock으로 보호하고, 이후 실제 조회는 read lock으로 처리했습니다.
항상 read lock만 쓰면 초기화 과정에서 충돌 위험이 있고, 반대로 전부 write lock으로 처리하면 사실상 싱글 스레드처럼 동작해 성능 이점이 사라지기 때문입니다.

스레드 풀과 락 정책이 정상적으로 동작해서 정합성을 유지하면서도 안정적으로 처리할 수 있음을 확인했습니다.

# Step 11. 시행착오와 배운 점

마지막으로 시행착오를 말씀드리겠습니다.

처음에는 worker 수를 늘릴수록 처리량도 같이 올라갈 것이라고 생각했습니다.
하지만 실제 benchmark에서는 worker를 늘려도 항상 더 빨라지지 않았고, 오히려 더 느린 경우도 많았습니다.

원인은 lock과 queue, context switch 오버헤드였습니다.
저희 구조에서는 INSERT가 write lock을 사용하고, SELECT도 preload 단계에서 write lock을 거치기 때문에
worker 수가 많아질수록 병렬성보다 오히려 경쟁 비용이 더 크게 드러났습니다.

이 실험을 통해 멀티스레딩이 항상 더 빠른 것은 아니고,
thread 수보다 공유 상태와 lock 구조가 성능에 더 큰 영향을 준다는 점을 확인했습니다.

추가로 lock 범위를 더 줄이는 실험도 해봤고, 속도는 빨라졌습니다.
